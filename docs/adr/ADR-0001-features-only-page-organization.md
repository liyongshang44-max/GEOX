# ADR-0001: 页面组织只允许 features

- 状态：Accepted
- 日期：2026-04-06
- 决策者：Web 前端团队

## 背景

当前 `apps/web/src` 同时存在两套页面组织：

1. 新架构：`src/features/*/pages/*`
2. 旧架构：`src/views/*`

随着路由逐步模块化，继续在非 legacy 路由中直接引用 `views/*` 会带来以下问题：

- 页面归属不清晰，影响代码所有权与演进节奏；
- 新旧结构混用，增加维护成本与认知负担；
- 难以在 CI 中稳定执行“按 feature 组织”的约束。

## 决策

1. **页面组织规范**：新增页面仅允许落在 `src/features/*/pages/*`。
2. **legacy 隔离**：`src/views/*` 只用于 legacy 路由，并统一挂载在 `/legacy/*` 前缀下。
3. **兼容策略**：保留旧路径，通过重定向跳转到对应 `/legacy/*` 路径，避免历史链接失效。
4. **CI 门禁**：在 CI 中执行边界校验，禁止非 legacy 路由渲染 `views/*` 页面。

## 影响

### 正向影响

- 路由与页面边界更明确，符合 feature-first 组织方式；
- legacy 页面影响范围被压缩，迁移路径可控；
- 通过 CI 提前阻断回归问题。

### 代价与限制

- 短期内需要维护 legacy 路由与兼容重定向；
- 路由变更时需同步更新边界检查脚本。

## 落地步骤

1. 将 legacy 页面统一挂载到 `/legacy/*`。
2. 将旧路径改为重定向到 `/legacy/*`。
3. 在 CI 中运行 views 边界检查脚本。
4. 新页面开发必须放在 `features/*/pages/*`，禁止新增 `views/*` 依赖。
