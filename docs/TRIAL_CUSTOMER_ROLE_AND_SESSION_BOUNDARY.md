# GEOX 试点客户角色与会话边界说明

状态：草案  
适用范围：commercial_v1 试点交付阶段  
文档性质：边界说明，不是营销材料

## 一、本文档说明什么

本文档只说明一件事：

当前 GEOX 在试点阶段，客户如何进入系统，系统当前如何建立会话，以及当前会话能力的真实边界在哪里。

本文档不把当前实现包装成完整账号体系，也不把试点 token 模式包装成企业级 IAM / SSO。

---

## 二、当前仓库里的真实进入方式

当前仓库中，正式进入路径是：

- 前端登录页：`/login`
- 登录动作：前端调用 `POST /api/v1/auth/login`
- 登录成功后：前端进入 `/dashboard`

当前登录页要求用户输入或粘贴 access token。  
前端不会再要求客户手动修改 localStorage，也不会要求客户自己写 tenant / project / group。

当前登录成功后，前端依据后端返回结果建立会话，使用的最小字段包括：

- token
- tenant_id
- project_id
- group_id
- role
- scopes
- actor_id
- token_id

这是一套“试点阶段最小正式进入方式”，不是用户名密码体系，也不是多用户账号系统。

---

## 三、当前后端 auth 能力边界

当前后端 auth 是最小壳能力，不是完整身份平台。

仓库中当前已存在的 auth 路由包括：

- `GET /api/v1/auth/me`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/providers`

其中当前试点主路径实际使用的是 token 登录。

后端 `auth_v1` 的职责是：

- 校验 token 是否存在
- 校验 token 是否有效
- 返回 token 绑定的上下文与角色
- 提供最小 logout 壳接口

当前后端没有提供以下能力：

- 用户名/密码登录
- 用户注册
- 用户找回密码
- 组织/成员管理
- 企业 SSO
- SCIM
- 完整权限中心

因此，当前 auth 应被理解为：

> 试点交付阶段可用的最小身份壳能力

而不是：

> 完整 SaaS 身份系统

---

## 四、当前前端会话模型

当前前端会话是正式建立的，不再是单纯“工程 token 临时读一下”。

前端当前会话由以下对象组成：

- token
- context
  - tenant_id
  - project_id
  - group_id
- meta
  - role
  - scopes
  - actor_id
  - token_id

前端已存在独立的 `sessionModel`，并由 `AuthProvider` 管理。

### 4.1 会话建立

登录成功后：

1. 前端调用 `/api/v1/auth/login`
2. 检查返回结果中是否包含：
   - tenant_id
   - project_id
   - group_id
3. 若缺少任一上下文字段，则阻断进入
4. 若完整，则建立 session，并跳转到 `/dashboard`

### 4.2 当前不再允许的旧行为

当前仓库已经去掉以下旧行为：

- 不再默认使用 `t_demo / p_demo / g_demo`
- 不再允许“上下文缺失但照样进系统”
- 不再把 session 简化成“只存一个 token”

### 4.3 上下文冲突处理

如果浏览器里已有旧 session，而新登录返回的是另一套 tenant / project / group，上下文不一致时：

- 系统采用新登录上下文覆盖旧上下文
- 前端会给出提示，说明历史会话上下文与新登录上下文不一致，系统已切换到新上下文

当前仓库没有提供“手动切换租户/项目/组”的正式能力。  
上下文以当前登录 token 所绑定的结果为准。

## 五、当前路由门禁行为

当前前端正式门禁由 `RequireSession` 承担。

除 `/login` 外，其余业务主路径通过统一门禁检查后才进入应用壳层。

门禁检查逻辑包括：

- 本地是否存在已登录 session
- token 是否存在
- 调用 `/api/v1/auth/me` 复核当前身份是否仍有效
- 根据结果决定进入系统，或跳回 `/login`

这意味着当前系统已经具备正式的“进入前校验”行为，而不是“页面先进去，接口再随机报错”。

---

## 六、当前异常与会话失效处理

当前仓库已把以下情况纳入正式处理范围：

- `AUTH_MISSING`
- `AUTH_INVALID`
- `AUTH_REVOKED`
- `AUTH_SCOPE_DENIED`
- `AUTH_ROLE_DENIED`
- `SERVICE_UNAVAILABLE`

### 6.1 会话失效

以下情况视为会话失效：

- 没有登录态
- token 无效
- token 被撤销

当前系统行为：

- 清除本地 session
- 跳回 `/login`
- 给出正式提示，而不是只在控制台报错

### 6.2 权限不足

如果后端返回 scope 或 role 不满足：

- 当前前端会提示“当前身份仅允许查看/需联系实施或支持”
- 不再把权限不足混同为“token 失效”

### 6.3 服务不可达

如果接口超时、网络失败或服务异常：

- 当前前端会提示“服务暂不可用，请稍后重试”
- 不应理解为当前身份本身失效

---

## 七、当前退出边界

当前仓库顶部已提供退出入口。

但当前退出的真实行为是：

> 清除本地会话，然后回到 `/login`

顶部按钮当前文案已经明确写为：

> 退出（本地会话）

这表示当前退出边界是明确的：

- 当前主要实现的是本地 session 清理
- 当前前端没有把退出包装成“完整企业级登出”
- 当前不应向现场宣称已完成 external idp 联动退出

因此，当前 logout 的正确理解是：

> 试点阶段的本地会话退出能力

而不是：

> 完整企业身份系统的一体化登出能力

---

## 八、当前角色边界

当前仓库中，前端会读取 `role` 并展示，但当前系统不是完整角色中心。  
因此，本阶段角色说明应保持在“试点交付可解释”层级，而不是扩成复杂权限矩阵。

当前试点阶段，建议按以下三类身份理解：

### 8.1 客户查看

适用场景：

- 查看 dashboard
- 查看 field / operation / report 类页面
- 了解试点运行状态

边界：

- 不应承担实施初始化
- 不应承担调试与故障排查
- 不应默认拥有执行性管理动作

### 8.2 现场实施

适用场景：

- 使用试点 token 登录
- 确认当前 tenant / project / group 是否正确
- 按试点流程完成现场接入、验证与演示

边界：

- 负责试点运行与交付动作
- 不等同于完整系统管理员
- 不应把试点实现解释为完整 IAM 能力

### 8.3 技术支持

适用场景：

- 排查 session、权限、服务异常
- 处理登录失败、上下文冲突、服务不可达等问题
- 协助实施定位问题

边界：

- 可解释系统当前身份与会话机制
- 不代表系统已经具备完整企业身份平台

---

## 九、当前明确不支持的能力

基于当前仓库真实实现，以下能力不属于当前试点会话体系：

- 用户名 / 密码账号体系
- 用户自助账号管理
- 多租户切换台
- 完整用户中心
- SSO
- SCIM
- 企业级 IAM
- 复杂 RBAC / ABAC 权限系统
- 完整 external IdP 登出联动

---

## 十、现场使用时的正确表述

现场实施或技术支持在解释当前系统时，应使用以下口径：

可以说：

- 当前 GEOX 已提供正式登录入口
- 当前 GEOX 已提供最小试点会话机制
- 当前系统会明确识别 tenant / project / group 上下文
- 当前会话失效、权限不足、服务不可达都有正式前端反馈

不应说：

- 当前 GEOX 已具备完整账号体系
- 当前 GEOX 已完成企业级 IAM
- 当前 GEOX 已支持完整 SSO
- 当前 GEOX 已具备完整用户中心

---

## 十一、结论

当前 GEOX 身份与会话能力，已达到：

- 试点客户可进入
- 现场实施可解释
- 技术支持可排查

但当前能力边界仍然明确：

> 这是试点阶段的最小正式进入方式，不是完整身份平台。
