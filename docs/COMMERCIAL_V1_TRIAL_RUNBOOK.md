# GEOX commercial_v1 试点 Runbook

## 1. 目的与适用范围

本 Runbook 只用于 GEOX 当前 `commercial_v1` 试点交付场景，目标是把已经存在的系统能力收口成一套现场可直接执行的最小流程。

本 Runbook 适用的唯一主交付基线是：

- `docker-compose.commercial_v1.yml`

当前仓库中该 compose 已明确包含：

- postgres
- mqtt
- minio
- minio-init
- server
- telemetry-ingest
- jobs
- executor
- web

并明确了以下正式外部入口：

- server：`http://127.0.0.1:3001`
- web：`http://127.0.0.1:5173`
- postgres：`5433`
- MinIO：`9000`
- MinIO console：`9002`

同时，`GEOX_AO_ACT_TOKEN` 在 compose 中是强制必填；evidence bucket 由 `GEOX_EVIDENCE_S3_BUCKET` 控制，默认值为 `evidence-exports-v1`。

本 Runbook 不适用于旧的外部 `3000` 口径。`3000` 在当前仓库中是 server 容器内部端口，宿主机正式入口是 `3001`。

---

## 2. 角色分工

### 2.1 客户 / 销售

客户 / 销售不负责做技术初始化，也不负责排障。客户侧当天只需要围绕固定演示路径理解 GEOX 的价值：

- 首页：看整体经营入口
- 地块页：看单个地块是否已经进入“设备—数据—建议—作业”的经营链
- 作业页：看闭环，不看底层技术细节

### 2.2 现场实施

现场实施负责把试点环境按固定顺序拉起，并完成首日验证：

1. 启动 commercial_v1 环境
2. 检查 server / web
3. 创建 field
4. 接入 device
5. 初始化 program
6. 确认首条 telemetry
7. 确认建议 / 作业对象已具备演示条件
8. 带客户走固定演示路径

现场实施不需要翻代码，只需要按本 Runbook 和页面正式入口执行。

### 2.3 技术支持

技术支持只在以下场景介入：

- commercial_v1 没起起来
- 页面可开但对象不完整
- 设备离线
- 首条 telemetry 未到达
- program / recommendation / operation 无法承接

技术支持必须按本 Runbook 的最小排障顺序处理，不现场临时发明第二套流程。

## 3. 当前仓库中的正式页面入口

当前前端路由已明确以下正式入口：

- 首页：`/` 或 `/dashboard`
- 田块列表：`/fields`
- 新建田块：`/fields/new`
- 田块详情：`/fields/:fieldId`
- 设备列表：`/devices`
- 设备接入向导：`/devices/onboarding`
- 设备详情：`/devices/:deviceId`
- 作业列表：`/operations`
- 作业详情：`/operations/:operationPlanId`
- 经营方案列表：`/programs`
- 初始化经营：`/programs/create`
- 经营方案详情：`/programs/:programId`

其中首页默认就是 `CommercialDashboardPage`，不是临时调试页。

## 4. 试点当天的固定目标

试点当天的目标不是把所有历史能力都讲一遍，而是完成下面这条最小可演示路径：

1. 环境启动成功
2. web 可访问
3. field 已创建
4. device 已接入
5. program 已初始化
6. 首条 telemetry 已确认
7. 地块页可展示首日状态
8. 作业页可展示闭环对象或闭环承接入口

只要以上链路成立，就进入客户演示。  
若基础对象未齐，不要把“服务启动成功”误说成“业务闭环已可演示”。

---

## 5. 最短演示路径

最短演示路径固定为：

**首页 → 地块页 → 作业页**

这是第五组正式口径，不允许现场把设备接入页、legacy 调试页、admin 页当作客户主演示路径。

### 5.1 首页讲什么

当前首页路由落到 `CommercialDashboardPage`。

首页当天要讲的是：

- 这是 GEOX 的平台总览入口
- 后续所有试点对象从这里进入
- 不是单点设备页，而是经营视角入口

客户 / 销售在首页只讲“统一经营入口”，不讲容器、Topic、Token。  
现场实施在首页只确认页面能正常打开，不是白屏或报错。  
技术支持只在首页判断系统是否存在明显全局故障。

### 5.2 地块页讲什么

地块详情页是当前仓库里最接近“首日试点检查面”的页面。

`FieldDetailPage` 里已经直接内置了“首次数据可见性检查”，检查项包括：

- 田块是否已创建
- 是否已绑定设备
- 设备是否在线
- 是否收到首条数据
- 是否已有建议
- 是否已有作业

并且每一项都已经带了对应动作入口，例如：

- 去绑定设备
- 查看设备状态
- 查看接入说明
- 查看建议
- 查看作业

这说明地块页是试点当天判断“现场是否已形成经营对象链”的核心页面。

客户 / 销售在地块页要讲：

- 这个地块已经正式纳入系统
- 当前是否已挂设备
- 当前是否已看到首条数据
- 当前是否已经出现建议或作业链承接

现场实施在地块页只需要确认：

- field 已存在
- 设备是否已绑定
- 设备是否在线
- 是否已有首条 telemetry
- 是否已有建议状态
- 是否已有作业状态

技术支持在地块页用它判断问题属于哪一层：

- field 未创建
- device 未绑定
- device 未在线
- telemetry 未到达
- recommendation 未出现
- operation 未出现

### 5.3 作业页讲什么

当前前端正式作业路由是：

- `/operations`
- `/operations/:operationPlanId`

作业页当天要讲的是：

- GEOX 不只是看到地块状态
- 系统还能承接到作业对象
- 作业对象可以继续往执行、回执、证据、验收方向推进

客户 / 销售在作业页讲“闭环”。  
现场实施在作业页确认对象是否存在、详情是否能打开。  
技术支持在作业页确认是对象本身缺失，还是页面承接问题。

## 6. 现场实施固定执行顺序

试点当天必须按以下顺序执行，不允许跳序。

### Step 1：启动 commercial_v1 环境

唯一正式基线：

- `docker-compose.commercial_v1.yml`

启动前必须确认：

- `GEOX_AO_ACT_TOKEN` 已提供
- 现场知道 server 外部入口是 `3001`
- 现场知道 web 外部入口是 `5173`
- 现场知道 MinIO 是 `9000`
- 现场知道 MinIO console 是 `9002`
- evidence bucket 口径是 `GEOX_EVIDENCE_S3_BUCKET`
- 不再使用旧 `3000` 作为宿主机入口

完成标准：

- 环境启动
- server 对外可访问
- web 对外可访问

现场人员此时只需要确认：环境按 commercial_v1 起起来了，而不是混用了其他 compose。  
客户此时不看技术细节。  
技术支持此时只看基线是否一致。

### Step 2：检查首页与基础可访问性

首页入口：

- `/`
- `/dashboard`

完成标准：

- web 能打开
- 首页能进入
- 不是空白页、崩溃页、明显错误页

现场人员此时只需要确认：系统已能从首页进入。  
客户此时只看“系统是否可打开”。  
技术支持此时不进入对象级排障。

### Step 3：创建 field

正式入口：

- `/fields/new`

当前 `FieldCreatePage` 已明确两条试点口径：

1. 边界可后续补充；即使暂无 GeoJSON 也可先完成创建并继续开局流程。
2. 创建后会自动跳转到田块详情，并继续绑定设备、初始化经营方案。

创建成功后，当前代码会跳转到：

- `/fields/:fieldId?created=1`

完成标准：

- field 创建成功
- 成功进入对应田块详情页

现场人员此时只需要确认：田块已经建立，不要卡在边界未补齐。  
客户此时不看表单细节。  
技术支持只在创建失败时介入。

### Step 4：在 field detail 确认下一步

进入田块详情页后，当前页面已经给了两个层面的引导：

1. 页首主动作：
   - 查看经营方案 / 创建经营方案
   - 查看当前作业
   - 查看设备

2. 初始化 banner：
   如果当前田块尚未有经营方案，会出现“尚未完成初始化经营”的 banner，并给出：
   - 初始化经营
   - 去绑定设备
   - 返回田块列表

同时，该页还内置了首日检查清单。

现场实施在这个阶段要做的是：

- 确认是否已有绑定设备
- 若无，则进入设备接入 / 设备中心
- 确认是否已有经营方案
- 若无，则进入初始化经营页

现场人员此时只需要确认：下一步是补设备还是补经营方案。  
客户此时只看田块已经进入系统。  
技术支持此时开始判断断点位于 field、device 还是 program。

### Step 5：接入 device

正式入口：

- `/devices/onboarding`

当前 `DeviceOnboardingPage` 已经明确是“固定 6 步接入流程”，并且有基础上下文输入：

- token
- device ID
- 设备名称
- device_mode
- device_template
- 目标田块 ID

页面底部已存在三条正式后续动作：

- 跳转设备详情
- 返回设备列表
- 返回田块继续首日验证

完成标准：

- 设备接入流程已走通
- 可以进入设备详情，或返回田块继续首日验证

现场人员此时只需要确认：设备已进入系统，且能回到田块继续验证。  
客户此时不看接入参数。  
技术支持只在接入失败、回流失败时介入。

### Step 6：确认 device 当前状态

设备详情页当前已经能承接试点当天最关键的设备状态判断。

`DeviceDetailPage` 中当前已直接展示：

- 在线状态
- 绑定田块
- 最近命令
- 最近回执
- 绑定状态
- 首条数据状态

并且有正式动作入口：

- 查看绑定田块
- 查看作业中心

若设备离线，页面还会给出：

- 离线排查入口（`/devices/onboarding`）
- 查看最近作业

设备详情页还保留了“接入流程入口”，可回到 onboarding 向导继续执行或复盘。

现场实施在这一步要确认：

- 设备是否在线
- 是否绑定到了正确田块
- 首条数据状态是否已完成
- 如离线，是否要回接入向导排查

完成标准：

- 设备在线或至少已明确离线原因
- 设备已绑定或可以继续绑定
- 首条数据状态可判断，不再处于完全不可知状态

现场人员此时只需要确认：设备在线、已绑定、首条数据状态可判断。  
客户此时只看“设备已经挂上来并开始工作”。  
技术支持此时开始处理离线或绑定异常。

### Step 7：初始化 program

正式入口：

- `/programs/create`
- 推荐带 field：`/programs/create?field_id=<FIELD_ID>`

当前 `ProgramCreatePage` 已明确是“初始化经营”，不是复杂方案编辑页。  
页面文案已写明：

- 使用模板快速创建首个 Program
- 无需填写复杂农学策略字段
- 首轮只需填写田块、作物、目标产量和模板

创建成功后，当前代码会跳到：

- `/programs/:programId?created=1`

完成标准：

- program 创建成功
- 进入 ProgramDetail
- field 与 program 的关系已建立

现场人员此时只需要确认：经营方案已经初始化成功。  
客户此时只看“系统已经开始按经营对象组织”。  
技术支持只在 program 创建失败时介入。

### Step 8：在 program detail 确认回流路径

当前 `ProgramDetailPage` 中，若 query 含 `created=1`，会显示成功提示：

- “Program 已创建成功。建议按‘方案页 → 田块页 → 作业页’路径持续推进经营闭环。”

页面同时提供以下正式返回路径：

- 查看关联田块
- 查看当前作业
- 返回方案列表
- 返回总览

完成标准：

- program detail 能打开
- 现场知道创建后要从方案页回到田块页，再进入作业页

现场人员此时只需要确认：创建成功后，不需要再猜下一步去哪。  
客户此时不看方案内部细节。  
技术支持只在页面回流断链时介入。

### Step 9：确认首条 telemetry 与首日状态

当前仓库里，“首条 telemetry 是否到达”不能只看日志，优先看以下两个页面：

1. `FieldDetailPage`
   - 是否收到首条数据
   - 设备是否在线
   - 是否已有建议
   - 是否已有作业

2. `DeviceDetailPage`
   - 首条数据状态
   - 最近遥测
   - 在线状态

同时，首日判断除页面检查外，还必须参考：

- `apps/server/scripts/commercial_v1_trial_day1_check.mjs`

完成标准：

- field 页或 device 页上，首条数据状态已明确
- 不再处于“设备对象创建了，但系统仍完全没有首条数据承接”的状态
- 页面判断与 day1 check 脚本输出不冲突

现场人员此时只需要确认：首条数据已经到了，或者已经明确卡在哪。  
客户此时只看“系统已经看到真实数据”。  
技术支持此时才处理 telemetry 未到达问题。

### Step 10：确认建议 / 作业是否具备演示条件

当前 `FieldDetailPage` 已把以下两项纳入首日检查：

- 是否已有建议
- 是否已有作业

并给了相应动作：

- 查看建议 / 刷新评估
- 查看作业 / 创建或查看作业

因此，现场当天至少要确认以下之一：

- 田块页已经出现建议相关状态
- 田块页已经出现作业相关状态
- 可从 program detail 或 field detail 进入作业页

完成标准：

- 可以从首页 → 地块页 → 作业页走通
- 每一段都有真实对象承接，而不是空页面跳转

现场人员此时只需要确认：客户已经能看到建议或作业链承接。  
客户此时只看“系统不只是接入成功，而是进入了经营闭环”。  
技术支持只在对象未出现时介入。

## 7. 客户演示时每个页面讲什么

### 7.1 首页

首页只讲“统一经营入口”。

建议表达：

- 这是 GEOX 的平台总览入口
- 后续所有试点对象从这里进入
- 不是单点设备页，而是经营视角入口

不要讲：

- compose
- topic
- token
- 容器端口细节

### 7.2 地块页

地块页讲“单个地块已经进入经营链”。

建议表达：

- 这个地块已被正式创建
- 设备是否已挂上来
- 设备是否在线
- 首条数据是否已经到了
- 当前是否已有建议或作业承接

这页是客户最容易理解 GEOX“从设备到经营”的页面。

### 7.3 作业页

作业页讲“闭环”。

建议表达：

- GEOX 不只识别状态，还能承接作业对象
- 作业页是闭环入口
- 后续可继续往执行、回执、证据、验收方向推进

这页是把“看板系统”和“经营闭环系统”区分开的关键页。

## 8. 最小排障顺序

出现问题时，必须按以下顺序排障，不允许跳来跳去。

### 8.1 先确认是不是按 commercial_v1 启动

先核对：

- 是否使用 `docker-compose.commercial_v1.yml`
- 是否误用其他 compose
- 是否还在把 `3000` 当宿主机入口

当前仓库中，server 容器内部端口是 `3000`，正式宿主机入口是 `3001`。

### 8.2 再确认 3001 / 5173 是否通

先确认：

- `http://127.0.0.1:3001`
- `http://127.0.0.1:5173`

如果这两层都不通，不进入业务对象排查。

### 8.3 再确认 token 是否正确

compose 中当前对 `GEOX_AO_ACT_TOKEN` 是强制必填。  
若 token 不对，后续页面和 API 不可信。

### 8.4 再确认 field / device / program 是否存在

不要一上来就怀疑 recommendation 或 operation。

先确认：

- field 是否已创建
- device 是否已接入
- program 是否已初始化

### 8.5 再确认设备是否在线

优先看：

- `FieldDetailPage` 中“设备是否在线”
- `DeviceDetailPage` 中在线状态

若设备离线，优先回设备接入页，不要直接声称“系统坏了”。

### 8.6 再确认首条 telemetry 是否到达

优先看：

- `FieldDetailPage` 中“是否收到首条数据”
- `DeviceDetailPage` 中“首条数据状态”
- `apps/server/scripts/commercial_v1_trial_day1_check.mjs` 输出

首条数据没到时，只能算“设备已接入未闭环”，不能算“首日可演示”。

### 8.7 最后确认建议 / 作业对象是否出现

只有当前面都成立后，才看：

- 是否已有建议
- 是否已有作业
- 作业页是否可承接

## 9. 当天完成定义

### 9.1 客户 / 销售完成定义

当天客户 / 销售只要能看懂以下三点，就算成功：

- 首页是统一经营入口
- 地块页已承接设备、数据、建议、作业首日状态
- 作业页代表系统已经具备闭环承接能力

### 9.2 现场实施完成定义

现场实施当天必须做到：

- environment 按 commercial_v1 起
- field 创建完成
- device 接入完成
- program 初始化完成
- 首条 telemetry 已确认
- 能带客户走完：首页 → 地块页 → 作业页

### 9.3 技术支持完成定义

技术支持当天必须做到：

- 问题出现时按固定顺序排障
- 不把现场问题变成临时读代码会
- 能快速判断问题属于：
  - 启动基线问题
  - token 问题
  - field / device / program 缺失
  - device 在线问题
  - 首条 telemetry 未到达
  - 建议 / 作业未承接

---

## 10. 当天不应做的事

试点当天不应做以下事情：

- 不切换到其他 compose 作为主交付
- 不再把旧 `3000` 当外部入口
- 不把 legacy / admin / dev 页面当客户主演示页

- 不在首条 telemetry 未确认前说“已经可以闭环演示”
- 不把工程日志、topic、容器状态当客户主观看内容
