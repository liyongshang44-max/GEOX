# COMMERCIAL_V1 试点主交付环境基线（第五组统一口径）

> 适用对象：交付团队、现场实施、运维值守、客户联调窗口。  
> 目的：把第五组试点的主交付环境冻结为**单一、不可歧义**的执行基线，避免现场因口径并行导致启动失败、验收偏差或责任不清。

---

## 1. 主交付基线（唯一）

第五组试点交付统一只认 **commercial_v1**，不再并列列出其他 compose 作为主口径。  
**唯一主 compose 路径：`docker-compose.commercial_v1.yml`**

> 现场执行、交付文档、验收记录、问题定位，均以该 compose 为唯一基线。

---

## 2. 启动前置变量（必须具备）

在启动 `docker-compose.commercial_v1.yml` 前，必须完成以下变量注入：

- `GEOX_AO_ACT_TOKEN`
- `GEOX_EVIDENCE_S3_BUCKET`
- `GEOX_EVIDENCE_S3_PUBLIC_ENDPOINT`

说明：以上为试点运行的前置条件；缺失任一项，均不应进入现场启动与验收流程。

---

## 3. 试点运行对外入口与端口口径（现场统一使用）

以下为试点运行实际必需的 host 入口口径：

- server 外部入口：`http://127.0.0.1:3001`
- web 外部入口：`http://127.0.0.1:5173`
- postgres 端口：`5433`
- MinIO：`9000`
- MinIO console：`9002`

> 对外联调、客户访问、测试脚本、验收清单必须统一以上口径，不得私自替换为其他端口。

---

## 4. Evidence Bucket 与 MinIO 口径

- Evidence 数据桶以 `GEOX_EVIDENCE_S3_BUCKET` 为准。
- Evidence 对外可访问地址口径以 `GEOX_EVIDENCE_S3_PUBLIC_ENDPOINT` 为准。
- MinIO 服务端口统一为 `9000`，控制台统一为 `9002`。

现场排障时，如出现“桶名不一致”“外链地址不可达”“控制台端口误用”，应首先核对本节三项口径是否被严格执行。

---

## 5. 关于 3000 端口的强制说明

**外部不再使用 3000，3000 仅为容器内端口。**

- 外部访问 server 统一走 `http://127.0.0.1:3001`。
- 文档、培训材料、脚本注释、口头说明中，均不得将 `3000` 作为外部入口。
- 发现旧口径（外部走 3000）时，按不符合基线处理并立即纠正。

---

## 6. 交付执行要求（面向现场）

1. 交付前检查：确认主 compose 文件路径唯一为 `docker-compose.commercial_v1.yml`。  
2. 启动前检查：确认 3 个前置变量已就绪。  
3. 联调前检查：确认外部入口统一为 `3001 / 5173`，数据库端口为 `5433`，MinIO 为 `9000 / 9002`。  
4. 验收记录：在交付记录中明确写入“commercial_v1 单一主口径”与“外部不使用 3000”。

本文件即第五组试点主交付环境冻结基线，后续现场执行按本文件为准。
