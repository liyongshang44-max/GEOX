# TWIN_KERNEL_RUNTIME_VALUE_TRACE_ACCEPTANCE

This task validates the missing runtime value layer after TK0–TK6 and TK-DATA-0.

It does not add a new business object, a new database table, a new seed dataset, or a new frontend page.

It verifies that the existing Twin Kernel builders can take fixed entered/collected fixture data and produce a complete, deterministic, explainable field-level twin trace.

## Problem being fixed

TK0–TK6 created the formal object skeleton:

```text
field_state_snapshot_v1
→ forecast_run_v1
→ scenario_set_v1
→ calibration_replay_v1
→ forecast_error_v1
→ field_learning_candidate_v1
→ decision_cycle_v1
```

TK-DATA-0 then fixed the provenance contract:

```text
entered / collected data
system-derived data
human-confirmed data
pointer refs
forbidden automatic writes
```

The remaining gap is runtime value proof: a field trace must show, in one run, what was provided as input and what was derived by the Twin Kernel.

## Acceptance scope

The acceptance script must run the actual TypeScript builders for TK1–TK6.

It must not use a database seed as proof. Seed data may exist separately, but this acceptance uses a fixed in-memory fixture so that provenance stays explicit and deterministic.

The script must produce a `twin_trace` output containing:

- fixture scope
- entered/collected source rows
- human-confirmed references
- pointer refs
- system-derived TK objects
- current state answer
- seven-day forecast answer
- no-action baseline and option scenarios
- predicted vs observed forecast error
- learning candidate boundary
- human-in-the-loop decision-cycle boundary

## Required runtime chain

The script must build:

1. `field_state_snapshot_v1`
2. `forecast_run_v1`
3. `scenario_set_v1`
4. `calibration_replay_v1`
5. `forecast_error_v1`
6. `field_learning_candidate_v1`
7. `decision_cycle_v1`

## Provenance expectations

### Entered / collected fixture data

The fixture may include:

- field source row
- water-state estimate row
- soil-moisture sensing window row
- weather forecast row
- post-irrigation observed payload
- evidence refs

These are treated as external input. The Twin Kernel may read them or reference them. It must not claim to have derived them.

### System-derived data

The following must be generated by builders:

- snapshot ID and hash
- state vector
- confidence object
- forecast run ID and hash
- seven-day forecast points
- risk timeline
- scenario set ID and hash
- no-action baseline
- option scenarios
- calibration replay ID and hash
- predicted vs observed alignment
- forecast error ID and hash
- error value and direction
- learning candidate ID and hash
- learning statement
- H58 gate status
- decision cycle ID and hash
- state machine
- human gate
- boundary flags

### Human-confirmed / pointer refs

The runtime trace may carry these only as refs:

- recommendation ID
- approval ID
- operation plan ID
- action task ID
- receipt ID
- acceptance ID
- post-irrigation verification ID

The Twin Kernel must not create them.

### Forbidden automatic writes

The trace must prove:

- `automatic_recommendation_created = false`
- `automatic_approval_created = false`
- `automatic_task_created = false`
- `automatic_receipt_created = false`
- `automatic_acceptance_created = false`
- `automatic_roi_created = false`
- `automatic_field_memory_created = false`
- `model_updated = false`

## Non-goals

This task does not:

- create a new seed
- require a local Postgres runtime
- write TK objects to DB
- add new routes
- add frontend pages
- write ROI
- write Field Memory
- create recommendations
- create approvals
- create tasks
- dispatch actions
- update models
