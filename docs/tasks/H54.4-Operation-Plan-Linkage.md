# H54.4 — Operation Plan Linkage

## Purpose

H54.4 proves that an approved recommendation-derived approval decision can enter the existing operation-plan creation path.

This task does not implement a new operation-plan service. It links the already-existing approval-decision output to the already-existing create-operation-plan route.

## Task-line position

```text
H53.2 WaterStressState
→ H53.3 Forecast
→ H53.3 ScenarioSet
→ H53.4 RecommendationCandidate
→ H54.0 Control Connector Preflight
→ H54.1 Recommendation Approval Request Connector Gate
→ H54.2 Actionable Irrigation Approval Adapter
→ H54.3 Approval Decision Linkage
→ H54.4 Operation Plan Linkage
```

## Runtime proof

H54.4 verifies the existing operation-plan runtime boundary:

```text
approval_decision_v1 APPROVED
+ approval_request_v1 APPROVED transition
→ POST /api/v1/operator/approval-decisions/:decision_id/create-operation-plan
→ operator_approval_decision_operation_plan_submission_v1
→ operation_plan_v1 CREATED
→ operation_plan_index_v1 upsert
```

## Existing route used

```text
POST /api/v1/operator/approval-decisions/:decision_id/create-operation-plan
```

## Allowed writes in runtime acceptance

- temporary probe `approval_decision_v1`
- temporary probe `approval_request_v1` APPROVED transition
- `operator_approval_decision_operation_plan_submission_v1`
- `operation_plan_v1`
- `operation_plan_index_v1` upsert

All probe facts and index rows must be cleaned up before the acceptance exits.

## Forbidden writes

- `operation_plan_transition_v1`
- `ao_act_task_v0`
- `ao_act_receipt_v1`
- `as_executed_record_v1`
- `evidence_artifact_v1`
- `acceptance_result_v1`
- `water_response_verification_v1`
- `roi_ledger_v1`
- `field_memory_v1`

## Boundary

H54.4 stops at `operation_plan_v1` in `CREATED` status.

It must not create AO-ACT tasks. AO-ACT linkage is a later task and requires a separate boundary decision.
