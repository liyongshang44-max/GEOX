0. 为什么存在 Constitution 目录

GEOX 的控制平面不是按“功能模块”治理，而是按语义污染风险治理。

因此必须把以下内容提前冻结：

哪些语义永远不能出现

哪些对象只能描述、不能判断

哪些能力只能在特定阶段之后引入

constitution/ 目录的作用，是把这些裁定从 Sprint 语境中拿走，
并放在工程不可绕过的位置。

1. 治理顺序总览（Order of Authority）

从上到下，权威递减；从左到右，时间顺序。

[ 不确定性聚合禁令 ]
            ↓
[ 不确定性命名宪法 ]
            ↓
[ 不确定性对象化 ]
            ↓
[ 行动前空间存在化 ]
            ↓
[ （未启动）控制裁决 ]
            ↓
[ 执行实例化与审计 ]


关键原则：

上层未冻结，下层不得启动。
上层禁止的语义，下层不得“偷偷复活”。

2. 当前已冻结的宪法文件（Authoritative）

以下文件 已冻结、已生效、不可绕过：

2.1 不确定性聚合禁令（最高优先级）

GEOX-UncertaintyAggregation-Prohibition.md

永久禁止：

不确定性评分

加权 / 平均

总体可信度 / readiness

基于不确定性的 gating / 排序

该文件 优先级高于一切模型、UI、调度与实现

一句话裁定：

系统允许你说“不知道什么”，
但永远不允许你把这些未知压缩成“可以行动”。

2.2 不确定性来源命名宪法

GEOX-UncertaintySource-Taxonomy-v0.md

冻结系统唯一合法的不确定性来源词表

只负责“叫什么”，不负责“多严重”

禁止强度、等级、排序、行动映射

作用：

在语言层面切断“判断偷渡”。

2.3 不确定性对象化（只读信封）

GEOX-UncertaintyEnvelope-v0.md

把“不知道什么 + 来自哪里”对象化

仅用于审计与回放

不触发、不排序、不前置、不聚合

作用：

把未知从解释文本中剥离出来，但不赋予用途。

2.4 合法行动空间的只读存在化

GEOX-PermissionSet-v0.md

描述在给定窗口与主体下：

哪些 AO 类别仍处于“可被考虑的空间”

不裁决、不推荐、不 gating、不任务化

不拥有 AO taxonomy 的定义权

作用：

在 Control 裁决之前，先把“行动空间”作为事实存在化。

4. 依赖与禁止回流（Hard Rule）

允许的方向只有：

Evidence / State
   → ProblemState
      → Uncertainty*
         → PermissionSet
            → （未来）Control Constitution / Kernel
               → AO-ACT Task / Receipt


明确禁止：

下游对象反向修正上游对象

利用 Uncertainty / PermissionSet 生成行动前置条件

因“实现方便”在中间层偷渡判断语义

7. 一句话版本（治理顺序）

先冻结“不能想什么”，
再冻结“可以怎么说”，
再冻结“能存在什么对象”，
最后才轮到“系统可以做什么”。
