# 试点首日验证检查表（现场版）

> 适用对象：交付经理、现场实施、值守同学、客户联调窗口。  
> 目标：把“环境起来了”和“当天可演示业务闭环”分开检查，避免误判。

---

## 先说清楚：两件事不是一回事

**“试点环境起来了”不等于“业务闭环能演示”。**

- 环境起来了：通常只代表服务进程已启动、页面能打开。
- 闭环能演示：代表从田块 → 设备 → 数据 → 建议 → 作业链路当天可走通并可展示。

现场汇报时请分两段汇报：
1) 环境状态；2) 闭环状态。不得混报。

---

## 本检查表适用的正式基线

本检查表只适用于 GEOX 当前试点正式基线：

- 主交付 compose：`docker-compose.commercial_v1.yml`
- server 外部入口：`http://127.0.0.1:3001`
- web 外部入口：`http://127.0.0.1:5173`

执行首日检查时，不再混用旧 `3000` 外部口径。`3000` 在当前环境中属于 server 容器内部端口，不作为现场统一访问入口。

---

## 首日固定检查对象（按顺序）

1. 服务已启动  
2. web 可访问  
3. field 已创建  
4. device 已接入  
5. device 在线  
6. 首条 telemetry 到达  
7. program 已创建  
8. field detail 可看到建议或待建议状态  
9. operations 可进入闭环链

---

## 1) 服务已启动

**在哪里看**
- 现场启动终端（`docker-compose.commercial_v1.yml` 启动窗口）。
- server 外部入口：`http://127.0.0.1:3001`
- 如已接入监控，看服务健康页/状态页。

**看见什么算通过**
- 核心服务无持续重启，状态稳定。
- server、web、数据库/对象存储相关容器处于运行状态。
- `http://127.0.0.1:3001` 可访问。
- 现场使用的确实是 `docker-compose.commercial_v1.yml`。

**没通过先做什么**
1. 先核对本次试点基线 compose 与前置变量是否一致。  
2. 先处理启动即报错（变量缺失、端口冲突、镜像拉取失败）。  
3. 未恢复前，不进入业务链验证。

---

## 2) web 可访问

**在哪里看**
- 浏览器访问现场统一入口：`http://127.0.0.1:5173`
- 首页入口：`/` 或 `/dashboard`

**看见什么算通过**
- 页面能稳定打开，不白屏、不 5xx。
- 可进入田块、设备、经营方案等主导航页面。

**没通过先做什么**
1. 先确认 web 服务是否运行、端口是否正确。  
2. 再确认浏览器实际访问的是 `http://127.0.0.1:5173`。  
3. 仍失败则回到“服务已启动”排障。

---

## 3) field 已创建

**在哪里看**
- 入口：`/fields/new` 创建。  
- 创建后在 FieldDetail 或字段列表确认。

**看见什么算通过**
- 能创建成功并进入 FieldDetail（或列表可见该 field）。
- 田块对象存在，可作为后续设备接入与经营初始化的挂载点。

**没通过先做什么**
1. 先简化输入，至少保证 field_id 和名称有效。  
2. 地理边界可后补，不要因为边界暂缺阻塞首日流程。  
3. 创建失败则先确认接口返回错误并按字段修正。

---

## 4) device 已接入

**在哪里看**
- 入口：`/devices/onboarding`。
- DeviceDetail / DeviceList 可查看设备对象与绑定关系。

**看见什么算通过**
- 设备对象可见，且已完成与目标 field 的接入/绑定。
- 页面“后续动作”可继续回设备详情或回田块。

**没通过先做什么**
1. 先核对 device_id、令牌/接入参数。  
2. 重新执行接入步骤，不要跳过向导关键步骤。  
3. 必要时先用仿真模式打通首日演示链。

---

## 5) device 在线

**在哪里看**
- DeviceDetail 的在线状态字段。
- FieldDetail 中设备状态相关可见信息。

**看见什么算通过**
- 在线状态为 ONLINE（或页面等价“在线”标识）。
- 最近心跳时间持续更新（不是长期停在历史时间）。

**没通过先做什么**
1. 先查设备网络/电源/网关连通性。  
2. 再核对接入配置与目标环境地址。  
3. 在线未恢复前，不要误判“首条数据已到”。

---

## 6) 首条 telemetry 到达

> 这是“环境可用”到“业务可演示”的关键门槛。

**在哪里看（必须同时看页面口径 + 输出口径）**
- 页面侧：FieldDetail / DeviceDetail 的“最近心跳/最近数据/实时状态”区域。  
- 输出侧：`apps/server/scripts/commercial_v1_trial_day1_check.mjs` 输出（不是只看日志）。

**看见什么算通过**
- 页面侧出现“最近数据时间”或等价实时指标，且时间为当前时段。  
- 输出侧可拿到该设备/田块最新 telemetry 结果（对象存在、时间戳有效）。

**没通过先做什么**
1. 先确认设备在线是否已通过。  
2. 再检查采集周期、上报目标地址、设备时钟。  
3. 如页面无更新但脚本有数据，优先排查读模型/刷新链；如两侧都无数据，回到接入链排障。

> 注意：**不得只写“看日志”作为首条数据到达判定**。日志只能辅助，不能作为唯一验收口径。

---

## 7) program 已创建

**在哪里看**
- 入口：`/programs/create?field_id=...`。
- 创建后进入 ProgramDetail 页面。

**看见什么算通过**
- Program 对象创建成功，可进入对应 ProgramDetail。
- 页面显示创建成功反馈（created 标识触发的成功提示）。

**没通过先做什么**
1. 先确认 field_id 已正确传入并存在。  
2. 再核对必填项与模板选择。  
3. 创建失败时先按页面错误提示逐项修正。

---

## 8) FieldDetail 可看到“建议或待建议”状态

> 这是“建议已可生成”判断的现场口径。

**在哪里看**
- FieldDetail 的初始化清单/建议相关区域。  
- 推荐中心或作业对象列表（用于对象侧复核）。

**看见什么算通过**
满足以下任一即可判定“建议已可生成”：
1. FieldDetail 明确出现“已有建议”；  
2. FieldDetail 明确显示“待建议/可继续评估”的状态；  
3. recommendation/operation 已出现可追踪对象。

**没通过先做什么**
1. 先确认前置：首条 telemetry 已到达、program 已创建。  
2. 手动触发刷新评估（在 FieldDetail 或建议中心）。  
3. 若长时间无建议对象，排查规则配置与评估任务状态。

> 注意：**不得以“理论上可生成”作为通过依据**，必须有页面状态或对象存在作为证据。

---

## 9) operations 可进入闭环链

**在哪里看**
- FieldDetail 的“转为作业/查看作业”入口。  
- Operations 列表与详情页。

**看见什么算通过**
- 可以从建议侧进入作业链路（创建或进入 operation）。
- 作业对象存在且可查看状态/进展（形成“建议→作业”闭环路径）。

**没通过先做什么**
1. 先回看“建议可生成”是否已通过。  
2. 确认当前账号有作业访问/操作权限。  
3. 无对象时先回建议侧重新触发或重建演示样例。

---

## 首日联合判定口径

首日判断不能只看页面，也不能只看脚本。

必须同时参考：

- 页面可见状态
- `apps/server/scripts/commercial_v1_trial_day1_check.mjs` 输出

只有在下面条件同时成立时，才能判定“首日可演示”：

1. commercial_v1 环境已启动；  
2. 页面能进入正式入口；  
3. field / device / program 已具备；  
4. 首条 telemetry 已确认；  
5. 建议或作业链已有承接。

出现以下任一情况，都不能判定“首日可演示”：

- 只有环境启动，没有业务对象；
- 页面能打开，但脚本显示关键对象缺失；
- 脚本能跑通，但页面没有正式对象承接；
- 只有 device，没有首条 telemetry；
- 只有 telemetry，没有建议 / 作业承接。

---

## 首日对外汇报模板（建议照读）

- 环境状态：服务启动 / web 可访问 / 基础对象可创建。  
- 闭环状态：首条 telemetry 已到达（页面+脚本双证据）/ 建议已可生成（页面状态或对象证据）/ 已可进入作业链。  
- 若未通过：明确卡在哪一项、当前应急动作是什么、预计恢复时间。

本检查表用于首日现场“能否演示闭环”的统一判定，执行以本文件为准。
