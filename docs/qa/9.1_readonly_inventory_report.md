# 9.1 只读盘点回报（字段级 + 链路级 + 指标口径）

> 范围：仅基于当前代码仓库做只读盘点，不改动业务逻辑。
> 
> 对象：`/api/v1/actions/execute`、`/api/v1/dashboard/overview_v2`、`/api/v1/operations/:id/detail` 及前端调用链。

## 1) 字段级盘点（Contract Inventory）

### 1.1 执行入口 `POST /api/v1/actions/execute`

- 入参强校验：`tenant_id/project_id/group_id/operation_id/execution_plan` 为必填；`execution_plan` 内部字段包含 `action_type/target/parameters/execution_mode/safe_guard/failure_strategy/idempotency_key` 等。
- 关键守卫：
  - `action_type` 受 allowlist 约束（含 `FERTILIZE`、`CHECK_FIELD_STATUS`）。
  - `safe_guard.requires_approval=true` 时直接 `403 REQUIRES_APPROVAL`。
  - `device_capability_check.supported=false` 时直接 `400`。
  - `idempotency_key` 命中历史请求时返回 `idempotent=true`。
- 落库事实：
  - 写入 `ao_act_task_v0`。
  - 写入 `action_execution_request_v1`，并携带 `execution_trace` 初始状态。

### 1.2 Dashboard Top Actions（`overview_v2`）

- 输出字段覆盖前端 `DashboardTopActionItem` 所需：
  - 优先级：`priority_bucket/priority_score/priority_components/global_priority_score/global_priority_components`
  - 可执行：`execution_plan/execution_ready/execution_blockers/device_capability_check`
  - 解释：`reason/risk_if_not_execute/recommended_next_action`
  - 追踪：`execution_trace`
- 额外输出：`priority_adjustment_by_trend` 与 `trend_adjustment_policy`，便于解释排序来源。

### 1.3 Operation Detail（`/api/v1/operations/:id/detail`）

- 已输出“解释 + 价值 + 执行上下文”字段：
  - `explain.system/human`
  - `value_profile`
  - `recommended_next_action/execution_plan/execution_ready/execution_blockers/device_capability_check/execution_trace`
  - `execution_context(tenant/project/group)`
  - `sla_snapshot/sla_definition`

### 1.4 字段级风险点（只读识别）

1. **SLA 口径文案不一致**
   - `overview_v2` 的 `sla_definition.execution_denominator` 与 operation detail 的同名字段描述不一致，可能导致产品/运营对“成功率分母”理解分裂。
2. **idempotency_key 生成策略偏静态**
   - `overview_v2` Top Action 生成的 `idempotency_key` 形如 `operationId_actionType`，同一 operation/action 多次触发将命中幂等返回，默认适合“防重”，但不适合“强制重试新任务”场景。

## 2) 链路级盘点（End-to-End Flow）

### 2.1 Dashboard 一键执行链路

1. 前端 `CommercialDashboardPage` 读取 `overview_v2.top_actions`。
2. 用户点击“一键执行”，前端将 `tenant/project/group + operation_id + execution_plan` 透传到 `executeOperationAction`。
3. 后端 `/api/v1/actions/execute` 进行 tenant match、allowlist、审批守卫、能力守卫、幂等检查。
4. 通过后写入任务事实，返回 `act_task_id`。
5. 前端刷新 `overview_v2`，展示新的 `top_actions` 排序与状态。

### 2.2 Operation Detail 一键执行链路

1. 详情页读取 `execution_context + execution_plan + execution_ready`。
2. 用户点击“一键执行”，调用同一执行 API。
3. 执行后 `reload()` 详情，刷新 `execution_trace`、`receipt/acceptance` 等衍生状态。

### 2.3 链路级风险点（只读识别）

1. **Dashboard 与 Detail 双入口共用同一幂等键语义**：在“需重复触发同动作”的运营场景下，可能被误判为重复提交。
2. **能力校验来源混合**：客户端带 `device_capability_check`，服务端也可二次判定；若两侧策略版本漂移，会出现前端可点但后端拒绝（或反向）。

## 3) 指标口径盘点（Metrics Definition）

### 3.1 已落地口径（代码显式）

- `overview_v2.sla_definition`：
  - 执行分母：已进入执行阶段的 operation（文案版）。
  - 验收分母：已进入验收阶段/有 receipt 的 operation（文案版）。
  - 响应时长：task 下发到 receipt 回传。
- `operation detail.sla_definition`：
  - 执行分母：存在 task 且非 invalid execution。
  - 验收分母：存在 receipt。
  - 响应时长：`task.dispatched_at | execution_started_at -> receipt.execution_finished_at`，缺失回退 `receipt fact occurred_at`。

### 3.2 口径一致性结论

- **结论：当前“执行分母”存在文本级不一致**（overview 与 detail 的描述粒度不同），建议在 9.2 统一成单一 SSOT 文案 + 计算口径。
- 其余口径（验收分母、响应时长）总体一致，但 detail 描述更细，应作为标准描述候选。

## 4) 9.1 回报结论（只读）

- 字段链路已贯通：TopActions → Execute API → Facts → Detail/Overview 回读闭环具备。
- 主要待处理项集中在“**口径统一**”与“**幂等键重试策略**”两处，不属于 9.1 只读阶段改动范围。
- 建议进入 9.2 时优先收敛：
  1) SLA 文案与计算口径 SSOT；
  2) 幂等键策略分层（防重键 vs 强制重试键）；
  3) 能力校验策略单源化（服务端权威 + 前端仅展示）。
