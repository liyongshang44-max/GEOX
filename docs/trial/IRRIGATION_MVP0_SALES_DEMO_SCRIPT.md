# GEOX 智能灌溉 MVP-0 销售演示脚本

## 1. 文档定位

本文档用于 GEOX 智能灌溉 MVP-0 的对外销售演示。它不是技术排障文档，不是完整产品白皮书，也不是合同文本。

本文档继承以下既有试点交付口径：

- `docs/COMMERCIAL_V1_TRIAL_RUNBOOK.md`
- `docs/TRIAL_DAY1_CHECKLIST.md`
- `docs/TRIAL_COMMERCIAL_BOUNDARY_AND_DELIVERY_SCOPE.md`
- `docs/TRIAL_PAYMENT_INVOICE_ACCEPTANCE_SOP.md`
- `docs/TRIAL_BILLING_AND_ACCEPTANCE_MIN_MODEL.md`

本演示只服务于一个销售目标：让客户在 15 分钟内理解 GEOX 如何把一次缺水事件转化为可审批、可执行、可验收、可记忆、可复盘的智能灌溉闭环。

---

## 2. 演示产品名称

**GEOX 智能灌溉付费共创试点包**

对外表达时不得称为：

- 成熟 SaaS 年费产品
- 全自动农业控制系统
- 全场景农业平台
- 增产承诺包
- 完整设备控制平台

---

## 3. 演示前提

演示必须基于 MVP-0 固定对象链：

1. 一个 demo field
2. 一条缺水 observation
3. 一个 irrigation recommendation
4. 一个 irrigation prescription
5. 一次人工 approval
6. 一个 mock valve skill run
7. 一个 receipt
8. 一个 as-executed
9. 一个 post-irrigation observation
10. 一个 acceptance
11. 三条 Field Memory
12. 四类 ROI 或其中适用项
13. 一个客户报告

如果链路中缺少 Field Memory 或 ROI Ledger，不得进入正式销售演示，只能说明当前为内部联调状态。

---

## 4. 15 分钟标准演示结构

### 0:00 - 1:00 客户痛点

要讲清楚三个问题：

- 缺水风险发现不及时，容易依赖人工跑田。
- 作业执行过程不透明，审批、执行、回执、验收经常分散。
- 试点价值难证明，客户和投资人很难看到节水、少跑田、提前预警等可解释收益。

建议话术：

> 今天演示的不是一个普通看板，而是一条智能灌溉闭环。我们从一块地的缺水信号开始，走到建议、处方、审批、执行、回执、验收、Field Memory 和 ROI Ledger，最后生成客户能看懂的报告。

不要讲：

- 容器、端口、topic、token
- 内部表名和调试对象
- 多场景宏大愿景

---

### 1:00 - 3:00 GEOX 如何发现缺水

演示对象：demo field 的缺水 observation。

要讲：

- 系统接收到土壤湿度数据。
- 该数据作为 observation 进入 sensing / evidence 判断。
- 如果数据过期或证据不足，系统不会强行进入成功链路。

建议话术：

> 系统先确认这块地是否真的出现缺水风险。不是只看一个页面状态，而是把传感器数据作为 observation，并进入证据判断。证据不足时，系统会要求人工复核，而不是直接下发任务。

客户应看到：

- 当前地块
- 当前缺水信号
- 证据是否足够
- 缺水风险触发原因

---

### 3:00 - 5:00 如何生成建议和处方

演示对象：irrigation recommendation 与 irrigation prescription。

要讲：

- Recommendation 是系统建议，不等于任务。
- Prescription 是正式处方合同，承接目标湿度、执行窗口、计划水量、风险等级、验收条件。
- Recommendation 不能绕过 Prescription 直接进入执行。

建议话术：

> 系统先形成灌溉建议，说明为什么建议灌溉。随后建议会被转成处方，处方才是可审批、可执行、可验收的正式对象。这样可以避免建议直接变成设备控制命令。

客户应看到：

- 建议原因
- 处方目标
- 风险等级
- 执行窗口
- 验收条件

---

### 5:00 - 7:00 如何审批

演示对象：approval。

要讲：

- MVP-0 默认人工审批。
- 高风险或证据不足场景不能自动执行。
- 审批结果会进入后续报告。

建议话术：

> GEOX 当前不是无人值守控制系统。处方进入执行前需要人工确认。审批留痕后，系统才允许进入任务和技能执行阶段。

客户应看到：

- 审批状态
- 审批人或审批角色
- 审批时间
- 审批备注

---

### 7:00 - 9:00 mock valve 如何执行并产生回执

演示对象：mock valve skill run、receipt、as-executed。

要讲：

- MVP-0 使用 mock valve 证明任务—Skill—执行—回执链路。
- 当前不承诺兼容所有真实阀门。
- 后续接真实设备时，必须通过 Skill 层声明能力、风险和审计。

建议话术：

> 这里我们用 mock valve 模拟阀门执行。重点不是证明今天能控制所有阀门，而是证明 GEOX 的任务、Skill、执行、回执、as-executed 链路是稳定可交付的。

客户应看到：

- 任务已生成
- mock valve skill run 已执行
- receipt 已产生
- as-executed 已记录实际执行量和执行时间

---

### 9:00 - 11:00 如何验收

演示对象：post-irrigation observation 与 acceptance。

要讲：

- 灌后数据用于判断作业效果。
- 验收不是只看设备说“已执行”，还要看灌后湿度是否回升、证据是否充分。
- 如果灌后没有响应，系统应进入失败或复核路径。

建议话术：

> 执行完成不等于验收通过。系统会比较灌前和灌后状态，并结合回执和证据判断本次灌溉是否达到预期。

客户应看到：

- 灌后 observation
- 验收状态
- 证据是否充分
- 是否需要人工复核或返工

---

### 11:00 - 13:00 Field Memory 记住了什么

演示对象：客户报告中的“系统记住了什么”。

必须展示三类记忆：

- 地块响应记忆：灌前湿度、灌后湿度、湿度变化、目标区间判断。
- 设备可靠性记忆：阀门响应、超时、回执完整性。
- Skill 表现记忆：缺水诊断是否被采纳、执行后是否验收通过。

建议话术：

> Field Memory 相当于这块地的长期记录本。系统不只是记录做了什么，还会记住这块地对灌溉的响应、设备是否可靠、这次判断是否被验证。下一次报告和建议会参考这些记忆。

注意：

- 第一版不承诺自动调参。
- 第一版只承诺 recommendation / report 可以读取上一轮 Field Memory。

---

### 13:00 - 15:00 ROI 怎么算，可信度多高

演示对象：客户报告中的“本次价值账本”。

MVP-0 只讲四类 ROI：

- 节水
- 少跑田 / 人工节省
- 异常提前发现
- 验收一次通过率

建议话术：

> ROI Ledger 会说明和什么比、怎么算、证据是什么、可信度多高。早期我们不承诺增产，因为增产需要至少一个作物季的数据。MVP-0 先验证节水、少跑田、异常提前发现和作业有证据、有验收。

必须明确：

- `MEASURED` 是实测值。
- `ESTIMATED` 是估算值。
- `ASSUMPTION_BASED` 是基于默认假设的估算。
- 低可信度 ROI 只能作为参考，不作为收益承诺。

## 6. 客户常见问题回答

### Q1：这是不是一个成熟 SaaS？

不是。当前销售的是“智能灌溉付费共创试点包”，采用项目制试点交付，不是公开 SaaS 年费订阅。

### Q2：为什么用 mock valve，不直接接真实阀门？

MVP-0 先验证任务—Skill—执行—回执—验收—ROI 主链。真实阀门接入属于后续试点扩展项，需要按设备能力、风险、安全边界单独确认。

### Q3：能不能承诺增产？

不能。增产是季末或多季指标。MVP-0 只验证节水、少跑田、异常提前发现和作业留痕。

### Q4：ROI 是实测还是估算？

报告会标注。没有 evidence_refs 的 ROI 不应标为实测；基于默认假设的 ROI 只能作为估算。

### Q5：试点失败怎么办？

试点复盘会根据 ROI Ledger 和 Field Memory 给出继续、调整或停止建议。失败原因可能来自感知层、设备、Skill 判断、执行链、证据不足或客户 baseline 缺失。

## 7. 演示完成标准

一次合格销售演示必须让客户理解：

1. GEOX 如何发现缺水。
2. GEOX 如何生成建议和处方。
3. GEOX 如何通过人工审批控制风险。
4. GEOX 如何通过 mock valve 证明执行链。
5. GEOX 如何通过灌后 observation 和 acceptance 判断效果。
6. GEOX 如何用 Field Memory 记录地块、设备和 Skill 表现。
7. GEOX 如何用 ROI Ledger 给出带基准线和可信度的价值证明。

如果客户只理解“这是个看板”，演示失败。  
如果客户误以为“这是全自动农业控制系统”，演示失败。  
如果客户理解“这是可收费共创试点包”，演示通过。
