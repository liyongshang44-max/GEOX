## 0. 目的

本文用于在 H52 前端收口之前，盘点当前前端路由表面。

本文回答以下问题：

1. 当前存在哪些路由。
2. 每条路由或导航入口由哪些文件拥有。
3. 每条路由属于哪个产品表面：customer、operator、admin、dev 或 legacy。
4. 哪些路由和文件存在混乱、重复、遮蔽、覆盖或死分支。
5. 每一类对象应采取什么处理动作：`KEEP`、`MIGRATE`、`MODIFY`、`LEGACY_VISIBLE_BY_URL_ONLY`、`DELETE_CANDIDATE`、`NEEDS_REVIEW`、`BUG_RISK / ACCEPTANCE_REQUIRED`。

本文不修改代码。本文是 H52 前端收口的规划基线。

## 1. 处理动作词汇表

| 动作 | 含义 |
| --- | --- |
| `KEEP` | 保留该路由或文件，作为未来产品表面的一部分，只做最小调整。 |
| `MODIFY` | 保留文件，但修改命名、导航、布局、路由归属或页面所有权。 |
| `MIGRATE` | 将能力迁移到新的 H52 路由结构中。不得立即删除。 |
| `LEGACY_VISIBLE_BY_URL_ONLY` | 保留路由，但只能通过直接 URL 访问，不进入正式导航。 |
| `DELETE_CANDIDATE` | 迁移、反向引用检查和验收覆盖之后，候选删除。 |
| `NEEDS_REVIEW` | 在行为和依赖确认前不得修改。 |
| `BUG_RISK / ACCEPTANCE_REQUIRED` | 路由行为可能与参数契约冲突，必须补验收后处理。 |

## 2. 重复路由四分类

当前仓库里“重复路由”不是一种问题，而是四种不同问题。H52 处理时必须区分。

### 2.1 Runtime duplicate

定义：同一路径在运行时可能由两套路由系统竞争匹配，最终渲染取决于 router 层级、catch-all 顺序或 layout 内部替换。

风险：最高。

处理：必须先写 route behavior acceptance，再修改。

### 2.2 Shadowed duplicate

定义：某个 route module 定义了路径，但更外层 shell 已经接管该路径，导致该 module route 很可能永远不会被触达。

风险：中高。

处理：先确认是否真的 shadowed，再迁移或删除候选。

### 2.3 Layout override

定义：route table 传入 children，但 layout 根据 pathname 自己替换真实页面。

风险：高。

处理：layout 不应拥有页面替换逻辑。H52 应把页面归属还给 route table。

### 2.4 Dead / stranded route

定义：route function 或 route declaration 存在，但没有被当前 router tree 调用或挂载。

风险：中。

处理：先做引用检查，确认无入口后标记 legacy 或 delete candidate。

## 3. 当前 Router 架构发现

### 3.1 Root App Router

主文件：

- `apps/web/src/app/App.tsx`

当前 root 行为：

- `/login` 渲染登录页。
- `/`、`/dashboard`、`/dashboard/customer`、`/dashboard/export` 重定向到 customer dashboard/export。
- `/customer/*` 使用 `CustomerShell`。
- `/admin/*` 使用 `AdminShell`。
- 其他路径进入 catch-all `Shell`，由 `AppShell` 和 `AppRoutes` 渲染。

问题：

当前 router 混合了三套路由系统：

1. 顶层 `/customer/*` 与 `/admin/*` shell。
2. catch-all legacy `Shell`，使用 `AppShell`、`SideNav`、`AppRoutes`。
3. 注入到 `AppRoutes` 的 route modules，其中不少仍定义 customer/admin/operator 路径。

影响：

路由所有权不清晰。部分 route module 仍定义已经被 CustomerShell 或 AdminShell 接管的路径。Operator 页面也同时存在旧 `/operator/workbench` generation 与新 `/operator/twin` generation。

处理意见：

`MODIFY`。

H52 应建立新的 canonical app shell，并把旧 shell routes 隔离。

## 5. H52 目标入口澄清

目标 canonical entrances：

```text
/app/operator
/app/customer
/app/admin
```

目标 operator routes：

```text
/app/operator
/app/operator/fields
/app/operator/fields/:fieldId/evidence-twin
/app/operator/fields/:fieldId/evidence-twin/water-stress
/app/operator/evidence
/app/operator/verifications
/app/operator/devices
/app/operator/settings
```

重要澄清：

这些 `/app/*` 路径是 H52 目标入口，不是当前 runtime 已实现入口。

当前运行时代码主要是：

```text
/customer/*
/admin/*
/operator/*
legacy catch-all routes
```

处理原则：

- 不得把 `/app/operator`、`/app/customer`、`/app/admin` 写成“当前可达入口”。
- 在 H52.1 IA 和 H52.2 实现前，它们只作为目标结构。
- 当前 routes 不应立即删除，应迁移或隔离。

## 12. H52 正式入口处理原则

### 12.1 Operator

目标：

```text
/app/operator
/app/operator/fields
/app/operator/fields/:fieldId/evidence-twin
/app/operator/fields/:fieldId/evidence-twin/water-stress
/app/operator/evidence
/app/operator/verifications
/app/operator/devices
/app/operator/settings
```

处理原则：

- Operator Twin pages 迁移到 Evidence Twin。
- Forecast / Scenario / Post-Irrigation 变成 Water Stress Loop 节点。
- ROI / Field Memory 不进入 Falcon 1 主线。
- Calibration 不是 P0。
- AO-SENSE / AO-ACT 页面必须保持边界：SenseTask 不等于 ActTask。

### 12.2 Customer

目标：

```text
/app/customer
/app/customer/fields
/app/customer/operations
/app/customer/reports
/app/customer/evidence-summary
```

处理原则：

- Customer 不展示 internal fact/projection/raw_payload/ao_act_task 等工程术语。
- Customer reports 消费结果，不解释内部执行链路细节。
- Customer layout override 必须先消除或验收确认。

### 12.3 Admin

目标：

```text
/app/admin
/app/admin/tenants
/app/admin/devices
/app/admin/imports
/app/admin/config
/app/admin/health
/app/admin/audit
/app/admin/skills
```

处理原则：

- Admin 独立于 old AppShell。
- Skill registry 和 run detail 应归 admin。
- Acceptance / import / health 页面可以 admin-only，但不进入 customer/operator。

## 13. 删除规则

不得直接删除以下类型文件：

- 已被 route table 引用的页面。
- 已被 CustomerLayout pathname override 引用的 views 页面。
- 已有 acceptance 覆盖的页面。
- Operator Twin pages。
- Customer report pages。
- Admin pages。

可以进入 `DELETE_CANDIDATE` 的前提：

1. 新 H52 route 已覆盖对应能力。
2. 旧 route 已从导航撤下。
3. 反向引用检查为 0。
4. 无 acceptance 依赖。
5. 有 legacy quarantine note。
6. 用户确认删除。
