## 0. 目的

本文定义 H52.1 Water Stress Loop 的前端验收标准。

Water Stress Loop 是 GEOX 的第一条农业产品闭环，也就是：

```text
水分压力闭环 / 猎鹰 1 号
```

本文只定义“怎样算前端验收通过”。

本文不要求新增后端接口，不要求新增事实类型，不要求新增数据库表，不要求创建新 seed，不要求实现写动作。

H52.1 P0 验收的核心是：

```text
前端必须真实展示水分压力闭环的证据链、状态链、建议候选链、执行后证据链和灌后验证链。

前端不得为了页面完整而造假。
前端不得打开写入口。
前端不得把候选、情景、预测、验证混成一个东西。
```

## 2. 验收对象

H52.1 Water Stress Loop P0 的正式验收对象是：

```text
/app/operator/fields/:fieldId/evidence-twin/water-stress
```

或在 H52 实现阶段的兼容 URL：

```text
/operator/twin/fields/:fieldId
/operator/twin/fields/:fieldId/forecast
/operator/twin/fields/:fieldId/scenarios
/operator/twin/fields/:fieldId/evidence
/operator/twin/fields/:fieldId/post-irrigation
```

但最终验收结论必须以目标产品结构为准：

```text
地块证据孪生 → 水分压力闭环
```

不是：

```text
Twin 总览 → Forecast / Scenario / Evidence / Post-Irrigation 多页面实验
```

## 3. 验收总原则

### 3.1 只做水分压力闭环

P0 只验收：

```text
SoilMoisture + CanopyTemperature + WeatherForecast + IrrigationEvent
→ WaterStressState
→ Forecast
→ Scenario
→ RecommendationCandidate
→ Approval
→ OperationPlan
→ AO-ACT
→ AsExecuted
→ Evidence
→ Acceptance
→ WaterResponseVerification
```

不验收：

- 氮素诊断。
- 病害诊断。
- 杂草诊断。
- 经济收益比较。
- 产量预测。
- Field Memory 写入。
- ROI 写入。
- Customer Report 生成。
- 自动审批。
- 自动派单。
- 自动 AO-ACT task 创建。

### 3.2 只读原则

H52.1 P0 页面必须只读。

页面允许：

- 读取。
- 展开。
- 追溯。
- 显示缺口。
- 显示候选。
- 显示执行后证据。
- 显示验证结果。

页面禁止：

- 创建 recommendation。
- 提交 scenario。
- 创建 approval_request。
- 作出 approval_decision。
- 创建 operation_plan。
- 创建 AO-ACT task。
- 提交 receipt。
- 写 evidence_artifact。
- 写 acceptance_result。
- 写 water_response_verification。
- 写 ROI。
- 写 Field Memory。
- 生成 customer report。

### 3.3 不造假原则

以下情况必须显示缺口，不得由前端补默认数据：

- `scenario.options = []`
- `no_action_baseline_present = false`
- `root_zone_soil_water_forecast_v1` 缺失
- `canopy_temperature_state` 缺失
- `weather_forecast_index_v1` 缺失
- `soil_moisture_sensing_window_v1` 缺失
- `water_response_verification_v1` 缺失
- `acceptance_result_v1` 缺失

### 3.4 证据优先原则

每个主要节点必须能展开看到：

- source refs
- evidence refs
- schema / code
- time
- quality
- confidence
- blocking reasons
- expand payload

没有 evidence_refs 的节点必须显示：

```text
证据引用缺失
```

不得静默隐藏。

## 5. 必须通过的 P0 验收项

### A. 页面定位验收

#### A-01 页面只展示水分压力闭环

验收标准：

- 页面标题或主标签必须包含 `水分压力闭环`。
- 页面可以显示副标题 `猎鹰 1 号`。
- 页面不得把氮素、病害、经济、产量作为主闭环节点。

通过条件：

```text
PASS: 页面主链只围绕水分压力。
FAIL: 页面把 nitrogen / disease / economics / yield 放入主 stepper。
```

#### A-02 页面必须位于地块证据孪生下

验收标准：

- 页面逻辑入口必须是 `fieldId`。
- 不得以 `operation_id`、`task_id`、`recommendation_id` 作为主入口。

通过条件：

```text
PASS: 主上下文是 field_id。
FAIL: 以 task_id 或 operation_id 作为页面主入口。
```

#### A-03 页面不得使用旧产品命名作为正式标签

禁止正式展示：

- Twin 总览
- 操作员 Twin
- Field Twin
- Operator Twin Workbench
- 地块 Twin 工作区

允许作为 legacy badge 展示：

```text
Legacy route: /operator/twin/...
```

---

### B. 原始信号验收

#### B-01 能展开 soil moisture 原始信号

验收标准：

页面必须能展示 soil moisture 的 RawSignal 来源，至少包括：

- metric
- value
- unit
- source_type
- source_id 或 fact_id
- occurred_at / captured_at / observed_at
- ingested_at 若存在

允许来源：

- `raw_telemetry_v1`
- `telemetry_index_v1`
- `raw_sample_v1`
- `facts`

失败条件：

- 只显示“土壤水分正常/异常”，没有来源。
- 只有图表，没有原始信号引用。

#### B-02 能区分文件 raw sample 与设备 telemetry

验收标准：

- 文件样本必须显示 `source_file` / `source_line_no` / `source_line_text`，若存在。
- 设备 telemetry 必须显示 `device_id`，若存在。

失败条件：

- 把 CAF raw sample 伪装成设备实时遥测。
- 把设备 telemetry 伪装成文件导入。

#### B-03 能展示 weather forecast 输入

验收标准：

页面必须能展示：

- `weather_forecast_fact_v1` 或 `weather_forecast_index_v1`
- forecast source / provider，若存在
- forecast time / generated time，若存在
- evidence refs

失败条件：

- 把 weather forecast 当作 GEOX root-zone forecast。
- 缺失 forecast 时自动生成假天气。

---

### C. 标准化观测验收

#### C-01 能展开 `device_observation_v1`

验收标准：

至少展示：

- observation_id
- metric
- value
- unit
- observed_at
- quality_flags
- confidence
- source_fact_id

失败条件：

- 只有 raw value，没有 observation_id。
- 前端自行判断观测质量，不显示后端 quality_flags。

#### C-02 能展开 `soil_moisture_sensing_window_v1`

验收标准：

至少展示：

- sensing_window_id
- window_start
- window_end
- expected_points
- actual_points
- coverage_ratio
- max_gap_ms
- quality_status
- source_observation_ids
- source_fact_ids

失败条件：

- 页面显示 soil moisture，但不显示窗口覆盖率。
- 覆盖率不足时不显示缺口。

#### C-03 观测缺失必须显示 gap

当以下任一对象缺失时，页面必须显示缺口：

- soil moisture observation
- soil moisture sensing window
- canopy temperature observation
- weather forecast input

不得静默降级。

---

### D. 水分压力状态验收

#### D-01 能看到 `water_state_estimate_v1` 或 `root_zone_soil_water_state_v1`

验收标准：

至少出现一个核心水分状态对象：

- `water_state_estimate_v1`
- `water_state_estimate_index_v1`
- `root_zone_soil_water_state_v1`

必须展示：

- state id / estimate id
- state code / state label
- confidence
- evidence_refs
- input refs

失败条件：

- 只显示“缺水/正常”，没有 state id 或 evidence refs。

#### D-02 canopy temperature 必须被处理

验收标准：

若存在：

- `canopy_temperature_state`
- `evapotranspiration_risk_state`

页面必须显示为水分压力辅助信号。

若不存在，页面必须显示：

```text
CANOPY_TEMPERATURE_NOT_IN_MAIN_LOOP
```

或等价 gap。

失败条件：

- 页面完全忽略 canopy temperature。
- 把 canopy temperature 缺失时伪装成正常。

#### D-03 WaterStressState 不得伪装成已存在事实类型

当前仓库尚未确认单独 `water_stress_state_v1`。

验收标准：

- 可以显示产品级 `WaterStressState` view。
- 必须注明它是 read model / composed state。
- 不得写成已落库 fact type，除非后续新增事实类型和迁移。

---

### E. Forecast 验收

#### E-01 能看到 root-zone forecast 或明确缺口

验收标准：

若存在 `root_zone_soil_water_forecast_v1`，必须显示：

- forecast_id
- source_state_id
- horizon_days
- daily_forecast / summary
- stress_day_count / limited_day_count 若存在
- confidence
- evidence_refs

若不存在，必须显示：

```text
ROOT_ZONE_SOIL_WATER_FORECAST_MISSING
```

或：

```text
SEVEN_DAY_FORECAST_RUN_MISSING
```

失败条件：

- 没有 forecast，却显示 7d forecast 正常。
- 把 `weather_forecast_index_v1` 当作 `root_zone_soil_water_forecast_v1`。

#### E-02 forecast 不得作为独立一级入口

验收标准：

Forecast 是 Water Stress Loop 的节点。

失败条件：

- 新导航中出现 Forecast 一级入口。
- Forecast 页面绕过地块证据孪生。

---

### F. Scenario 验收

#### F-01 能看到 scenario set 或明确缺口

验收标准：

若存在 scenario set，必须展示：

- scenario_set_id
- no_action_baseline_present
- options
- evidence_refs
- unavailable_reason

若不存在，必须显示：

```text
IRRIGATION_SCENARIO_SET_MISSING
```

#### F-02 options 为空时不得造假

验收标准：

若：

```text
options.length === 0
```

页面必须显示：

```text
SCENARIO_OPTIONS_MISSING
```

或等价缺口。

失败条件：

- 前端生成默认 10mm / 20mm / 30mm 情景。
- 前端用 mock option 填充页面。

#### F-03 no_action baseline 缺失时不得通过

验收标准：

若：

```text
no_action_baseline_present === false
```

Scenario 节点 status 必须为：

```text
LIMITED 或 MISSING 或 NOT_AVAILABLE
```

并显示：

```text
NO_ACTION_BASELINE_OR_OPTIONS_NOT_AVAILABLE
```

#### F-04 scenario 不能当作 task

页面必须显示或保留 boundary rule：

```text
SCENARIO_IS_NOT_TASK
```

失败条件：

- 点击 scenario 后直接创建 AO-ACT task。
- scenario option 被显示成已批准作业。

---

### G. Recommendation Candidate 验收

#### G-01 能看到 recommendation candidate

若存在 recommendation candidate，必须展示：

- recommendation_id
- selected_scenario_option_id，若存在
- action_type
- amount_mm
- evidence_refs
- human_approval_required
- no_direct_execution

#### G-02 必须显示人工审批要求

验收标准：

```text
human_approval_required = true
no_direct_execution = true
```

必须可见。

失败条件：

- 页面把 recommendation candidate 显示成已批准任务。
- 页面没有显示人工审批要求。

#### G-03 P0 不得提供提交 recommendation 写按钮

验收标准：

H52.1 P0 页面不得显示：

- 提交为建议候选
- Submit Scenario
- Submit Recommendation
- 创建建议
- 生成建议

处理意见：

- `SubmitScenarioToRecommendationPanel` 不进入 P0 正式 surface。
- 若组件保留，只能 legacy URL-only、dev-only 或 future write-enabled workflow。

失败条件：

- Water Stress Loop P0 页面出现该组件。
- Evidence Twin P0 页面可以触发 `submitOperatorScenarioRecommendation`。

---

### H. Approval / OperationPlan / AO-ACT 验收

#### H-01 能看到 approval / operation plan / AO-ACT / receipt / as-executed 链路

若 H31-H45 closure 中存在这些对象，页面必须展示：

- approval_request_v1
- approval_decision_v1
- operation_plan_v1
- operation_plan_transition_v1
- ao_act_task_v0
- ao_act_receipt_v1
- as_executed_record_v1

至少通过 stage group 或 execution tail 展示：

- task_id
- receipt_id
- as_executed_id

#### H-02 不得创建任何执行对象

页面不得：

- 创建 approval_request
- 作出 approval_decision
- 创建 operation_plan
- 创建 operation_plan_transition
- 创建 AO-ACT task
- 提交 receipt

必须显示 boundary：

```text
NO_AO_ACT_TASK_CREATION
NO_DISPATCH
NO_APPROVAL_BYPASS
```

---

### I. Evidence / Acceptance 验收

#### I-01 能看到 evidence artifact

若存在 `evidence_artifact_v1`，必须展示：

- artifact_id
- artifact_type
- source_kind
- subject_kind
- subject_ref_id
- pointer 或 summary
- evidence_refs

若不存在，显示 gap。

#### I-02 能看到 acceptance result

若存在 `acceptance_result_v1`，必须展示：

- acceptance_id
- task_id
- receipt_id
- as_executed_id
- status / verdict
- evidence_refs

若不存在，显示 gap。

#### I-03 acceptance 不能替代 verification

验收结果只能说明执行证据是否可接受。

它不能直接说明水分响应是否发生。

失败条件：

- 页面把 acceptance_result 显示为 water response verification。

---

### J. Water Response Verification 验收

#### J-01 能看到 water response verification

若存在 `water_response_verification_v1` 或 `water_response_verification_index_v1`，必须展示：

- verification_id
- pre_state_id
- post_state_id
- response_verdict
- class_transition
- available_water_fraction_delta
- weighted_matric_potential_kpa_delta
- confidence
- evidence_refs
- blocking_reasons

#### J-02 缺失 verification 时必须显示不可验证

若没有 water response verification，页面必须显示：

```text
WATER_RESPONSE_VERIFICATION_MISSING
```

或：

```text
NOT_VERIFIABLE
```

不得显示“已完成闭环”。

#### J-03 枚举不得混写

必须区分：

```text
Builder-level verdict enum:
RESPONDED / PARTIAL_RESPONSE / NO_RESPONSE_OBSERVED / NOT_VERIFIABLE / CONFLICTING_EVIDENCE

H51 demo display/index observed:
RESPONSE_OBSERVED / MODERATE_DEFICIT_TO_NORMAL
```

失败条件：

- 把 `MODERATE_DEFICIT_TO_NORMAL` 当 response verdict。
- 把 `RESPONSE_OBSERVED` 当 class transition。

```text
BUG_RISK / ACCEPTANCE_REQUIRED
```

## 9. 通过 / 不通过定义

### 9.1 通过

满足以下条件才算 P0 通过：

```text
页面只做水分压力闭环。
页面所有节点可追溯。
页面所有缺口明确显示。
页面所有写入口关闭。
页面没有伪造数据。
旧冲突页面已有处理意见且不污染 P0。
```

### 9.2 不通过

出现任一情况即不通过：

```text
前端造 scenario option。
前端造 forecast。
前端造 verification。
前端把 SubmitScenarioToRecommendationPanel 放进 P0。
前端能触发 recommendation submit。
前端能创建 AO-ACT task。
前端把 ROI / Field Memory 放入 Falcon 1。
页面以旧 Twin 总览作为正式入口。
页面没有 evidence_refs 或 source_refs 展开能力。
页面隐藏缺失数据。
```

## 11. 结论

Water Stress Loop Acceptance Spec 的核心不是让页面“能看”。

它的核心是防止三类错误：

1. 把不完整链路伪装成完整闭环。
2. 把只读证据页面变成写动作页面。
3. 把旧 Twin 实验页面继续当成新产品架构。

H52.1 P0 验收必须以这条原则为准：

```text
真实证据优先。
缺口必须显式。
写入口必须关闭。
旧页面必须降级。
```
