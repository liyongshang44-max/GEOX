## 0. 目的

本文是 H52.1 文档组进入 H52.2 实现前的实现护栏。

它不新增产品范围。

它不修改 H52.0 / H52.1 baseline 的语义。

它只冻结以下执行规则：

```text
实现前必须读基线文档。
不得私自扩散范围。
不得把目标接口误当作 P0 必须新增后端接口。
不得把 demo closure 暴露成正式 Evidence Twin contract。
不得伪造未暴露的数据。
不得让 P0 页面出现写入口。
```

任何 H52.2 相关实现任务开始前，必须先阅读并核对以下 6 份文档：

```text

docs/frontend-reset/H52.0-FRONTEND-ROUTE-INVENTORY.md

docs/frontend-reset/H52.1-WATER-STRESS-LOOP-ACCEPTANCE.md
```

阅读后必须确认：

1. 本次任务是否仍属于 Operator Evidence Twin / Water Stress Loop P0。
2. 是否触碰后端、seed、DB 或事实类型。
3. 是否触碰写入口。
4. 是否触碰旧 route 删除。
5. 是否引入 H52 文档未定义的新 loop、新页面或新动作。

如果答案不确定，必须暂停并提出问题，不得继续实现。

## 2. 不得私自扩散范围

H52.2 P0 只实现：

```text
操作员证据孪生
→ 地块证据孪生
```

不得私自扩散到：

- 氮素诊断。
- 病害诊断。
- 杂草诊断。
- 经济收益比较。
- 产量预测。
- 全季节经营优化。
- 自动审批。

- 自动创建 AO-ACT task。
- 自动写 Field Memory。
- 自动写 ROI。
- 自动生成客户报告。
- 模型自动学习。

如果实现中发现这些能力可能有价值，只能提出扩散建议，由项目负责人讨论确认后再建立新任务。

## 3. 目标 Evidence Twin API 不是 H52.2 P0 必须新增项

H52.1 View Model Contract 定义了目标接口：

```text
GET /api/v1/operator/fields/:fieldId/evidence-twin
GET /api/v1/operator/fields/:fieldId/evidence-twin?loop=water-stress
```

但 H52.2 P0 不强制新增该后端接口。

H52.2 P0 可以先通过前端 adapter 或后端 read model 聚合现有 read surfaces：

```text
GET /api/v1/operator/twin/source-indexes
GET /api/v1/operator/twin
GET /api/v1/operator/twin/fields/:fieldId
GET /api/v1/operator/twin/fields/:fieldId/forecast
GET /api/v1/operator/twin/fields/:fieldId/scenarios
GET /api/v1/operator/twin/fields/:fieldId/evidence
GET /api/v1/operator/twin/fields/:fieldId/post-irrigation
GET /api/v1/operator/twin/fields/:fieldId/h31-h45-closure
```

实现规则：

```text
不得因为 contract 存在目标 endpoint，就扩大范围强制新增后端 Evidence Twin API。
```

除非另有明确裁决，H52.2 P0 应优先使用 adapter 收口现有 read surfaces。

## 4. H31-H45 closure 是 adapter source，不是最终 contract

当前可用对象：

```text
operator_twin_h31_h45_closure_v1
```

它可以作为 H52.2 P0 的 adapter source。

它不得作为最终产品 contract 直接暴露。

H52.2 应将它映射到：

```text
water_stress_loop.steps
water_stress_loop.approval
water_stress_loop.operation
water_stress_loop.ao_act
water_stress_loop.as_executed
water_stress_loop.evidence
water_stress_loop.acceptance
water_stress_loop.verification
```

规则：

```text
operator_twin_h31_h45_closure_v1 是旧 Operator Twin demo closure projection。
H52 final view model 的 report_kind 必须是 OPERATOR_EVIDENCE_TWIN 或其明确后续版本。
不得把 OPERATOR_TWIN_H31_H45_DEMO_CLOSURE 当作正式 Evidence Twin report_kind。
```

## 5. 未暴露原始数据时必须显示 gap，不得造假

H52.0 Data Lineage Inventory 要求前端能够展示 RawSignal / Observation / StateEstimate / Evidence / Verification。

但 H52.2 P0 如果只消费现有 Operator Twin read surfaces，可能无法直接取得完整 raw telemetry 或 device observation payload。

因此，H52.2 P0 必须遵守：

```text
如果当前 read surface 未暴露 raw_telemetry_v1 或 device_observation_v1 的详情，页面必须显示 gap。
```

允许 gap code：

```text
RAW_SIGNAL_SOURCE_NOT_EXPOSED
OBSERVATION_SOURCE_NOT_EXPOSED
RAW_SAMPLE_DETAIL_NOT_EXPOSED
DEVICE_OBSERVATION_DETAIL_NOT_EXPOSED
```

这些 gap 表示：

```text
当前 P0 read surface 未暴露该层详情。
```

它们不表示：

```text
系统可以伪造该层数据。
```

禁止行为：

- 前端生成假 raw telemetry。
- 前端生成假 device observation。
- 前端把 source index 摘要伪装成完整 raw fact。
- 前端隐藏该层缺口。

## 6. string evidence_refs 必须规范化，不得丢弃

当前旧 read surfaces 中可能返回 string array 形式的 evidence refs，例如：

```json
["fact_123", "receipt_456", "acceptance_789"]
```

H52.1 View Model Contract 要求统一引用对象：

```ts
type EvidenceTwinRefV1 = {
  kind: string;
  ref_id: string;
  schema_ref: string | null;
  label: string | null;
  href: string | null;
};
```

H52.2 adapter 必须将 string ref 规范化为对象。

默认规则：

```json
{
  "kind": "external",
  "ref_id": "<raw_string>",
  "schema_ref": null,
  "label": "<raw_string>",
  "href": null
}
```

如果 adapter 能可靠识别 ref 类型，可以使用更具体的 kind：

```text
fact
index_row
receipt
as_executed
acceptance
verification
```

禁止行为：

- 丢弃 string evidence_refs。
- 只展示计数，不展示 ref。
- 将无法识别的 ref 强行归类为 fact。
- 生成无法验证的 href。

## 7. P0 页面不得导入或渲染写入口

H52.2 P0 是只读证据页面。

以下对象不得进入 H52.2 P0 页面：

```text
SubmitScenarioToRecommendationPanel
submitOperatorScenarioRecommendation
operator_scenario_recommendation_submission_v1 action surface
operator_root_zone_scenario_recommendation_submission_v1 action surface
```

H52.2 P0 页面不得出现以下交互：

- 提交为建议候选。
- Submit Scenario。
- Submit Recommendation。
- 创建建议。
- 生成建议。
- 创建 approval。
- 创建 operation plan。
- 创建 AO-ACT task。
- 派单。
- 提交 receipt。

静态检查要求：

```text
H52 P0 页面文件不得 import SubmitScenarioToRecommendationPanel。
H52 P0 页面文件不得 import submitOperatorScenarioRecommendation。
H52 P0 页面 DOM 不得出现“提交为建议候选”。
H52 P0 页面 DOM 不得出现“Submit Scenario”。
H52 P0 页面 DOM 不得出现“Submit Recommendation”。
```

## 8. Scenario 只迁移 read part

`OperatorFieldTwinScenarioComparePage.tsx` 当前包含两类能力：

1. 情景比较只读展示。
2. 提交情景为建议候选的写入口。

H52.2 P0 只迁移第一类。

可以迁移：

- Scenario status。
- No-action baseline 状态。
- Scenario options 表。
- Risk delta。
- Confidence。
- Failure conditions。
- Evidence refs。
- Unavailable reason。
- Boundary rule: `SCENARIO_IS_NOT_TASK`。

不得迁移：

- `SubmitScenarioToRecommendationPanel`。
- `submitOperatorScenarioRecommendation`。
- POST action。
- 任何创建 recommendation candidate 的按钮。

如果未来需要恢复写入链路，应单独建立 write-enabled workflow，不得混入 H52.2 P0 Evidence Twin。

---

## 9. P0 envelope 与 write policy 不变量

H52.2 P0 的任何 view model、adapter result 或页面级 envelope 必须满足：

```text
writeReady=false
dispatchReady=false
approvalReady=false
taskCreationReady=false
memoryWriteReady=false
roiWriteReady=false
```

所有节点的 write policy 必须满足：

```ts
write_policy: {
  write_ready: false,
  allowed_actions: []
}
```

禁止通过 UI、adapter 或 hidden handler 暴露写动作。

---

## 10. 旧 route 不得先删

H52.2 P0 不以删除旧页面为首要目标。

旧 route 处理顺序：

```text
MIGRATE
→ LEGACY_VISIBLE_BY_URL_ONLY
→ REDIRECT after acceptance
→ DELETE_CANDIDATE after dependency check
```

不得在第一个实现 PR 中直接删除：

- `OperatorFieldTwinWorkspacePage.tsx`
- `OperatorFieldTwinForecastPage.tsx`
- `OperatorFieldTwinScenarioComparePage.tsx`
- `OperatorFieldTwinEvidencePage.tsx`
- `OperatorFieldTwinCalibrationPage.tsx`
- `OperatorFieldTwinPostIrrigationPage.tsx`
- old `/operator/workbench` generation pages

Customer route governance 与 `/skills/runs` 参数问题仍是 H52 route governance P0，但不应阻塞 Operator Evidence Twin adapter 的只读实现，除非实现触碰相同 route owner。

## 11. 实现 PR 切分规则

H52.2 不应从大范围 React 重写开始。

推荐 PR 顺序：

```text
PR-1: H52.1-a implementation guardrails only
PR-2: Evidence Twin adapter types and normalization helpers
PR-3: Read-only Evidence Twin page skeleton
PR-4: Water Stress Loop stepper
PR-5: Scenario read-only section
PR-6: Post-irrigation verification section
PR-7: H52 Water Stress Loop acceptance script
PR-8+: route governance cleanup
```

每个 PR 必须声明：

- 读取了哪些 H52 baseline 文档。
- 是否触碰 backend / seed / DB / facts。
- 是否触碰 write action。
- 是否触碰 legacy route deletion。
- 是否存在范围扩散。

如果存在范围扩散，PR 不得继续，必须先提出讨论。

---

## 12. 完成条件

本 amendment 接受后，H52.2 可以进入第一个实现 PR，但必须满足：

1. 不新增后端接口作为 P0 前置条件。
2. 不新增事实类型。
3. 不新增 DB 表。
4. 不新增 seed。
5. 不删除旧页面。
6. 不打开写入口。
7. 不把 demo closure 当最终 contract。
8. 不伪造 raw signal / observation / forecast / scenario / verification。
9. 不丢弃 string evidence refs。
10. 不把未来扩展混入 P0。

---

## 13. 结论

H52.1-a 的作用是把 H52.2 从“前端改版”约束为“证据闭环只读实现”。

正式执行原则：

```text
每次任务先读文档。
只做文档定义范围。
缺口显式，不造假。
写入口关闭。
旧能力迁移，不先删除。
扩散先讨论，再决定。
```
