## 0. 目的

本文用于冻结 H52 前端重置后，操作员端的正式产品信息架构。

本文回答以下问题：

1. 操作员从哪里进入系统。
2. 操作员端一级导航是什么。
3. 地块证据孪生如何组织。
4. Water Stress Loop 放在哪里。
5. RawSignal / Observation / StateEstimate / Evidence / Verification 如何展开。
6. Forecast / Scenario / Post-Irrigation Verification 是主入口，还是链路节点。
7. 旧 Operator Twin / Field Twin / Workbench 页面如何迁移。
8. 哪些能力不进入 H52 操作员证据孪生主入口。

本文不改代码。本文是前端实现前的 IA 基线。

## 1. 总体产品定位

H52 操作员端不再以 `Twin 总览`、`双子总览`、`Operator Twin Workbench` 或旧 `Workbench` 作为正式产品入口。

H52 操作员端的正式定位是：

```text
操作员证据孪生
Operator Evidence Twin
```

一句话定义：

```text
面向操作员的地块证据孪生界面，用于查看一块地从原始信号到状态判断、建议候选、执行证据和验证结果的完整链路。
```

H52 操作员端首先服务一条农业闭环：

```text
Water Stress Loop v1 / 水分压力闭环 / 猎鹰 1 号
```

它不做万能农业平台，也不做全量客户交付门户，也不做后台治理控制台。

## 2. 当前仓库现状约束

### 2.1 当前 runtime 入口

当前 runtime 主要入口仍然是：

```text
/customer/*
/admin/*
/operator/*
legacy catch-all routes
```

当前 `App.tsx` 中已有 `/customer/*`、`/admin/*` shell，同时 `/operator/*` 还在 catch-all 旧 shell 下进入 operator routes。

### 2.2 当前 operator routes

当前 operator twin routes 是：

```text
/operator/twin
/operator/twin/fields/:fieldId
/operator/twin/fields/:fieldId/forecast
/operator/twin/fields/:fieldId/scenarios
/operator/twin/fields/:fieldId/evidence
/operator/twin/fields/:fieldId/calibration
/operator/twin/fields/:fieldId/post-irrigation
```

这些 route 当前可用，但不是 H52 的最终 canonical route。

### 2.3 当前 operator layout 命名问题

当前 `OperatorLayout.tsx` 仍使用：

```text
Twin 总览
GEOX 操作员 Twin
操作员数字孪生工作台
地块 Twin 工作区
```

这些命名应在 H52 实现阶段替换为：

```text
操作员证据孪生
地块证据孪生
水分压力闭环
```

### 2.4 当前 workspace 页面价值

当前 `OperatorFieldTwinWorkspacePage.tsx` 已经有可复用能力：

- scope policy
- current state vector
- data coverage matrix
- evidence summary
- data gap summary
- Fact / Estimate / Forecast / Scenario / Recommendation layer cards
- scenario boundary card
- recommendation candidate card
- H31-H45 decision-to-water-response chain card
- read-only boundary card

这些能力不应删除。H52 应迁移为 Evidence Twin 的正式节点和卡片。

## 3. H52 目标入口

H52 的目标 canonical entrances 是：

```text
/app/operator
/app/customer
/app/admin
```

注意：

```text
/app/* 是 H52 目标入口，不是当前 runtime 已实现入口。
```

H52 操作员端目标 route 是：

```text
/app/operator
/app/operator/fields
/app/operator/fields/:fieldId/evidence-twin
/app/operator/fields/:fieldId/evidence-twin/water-stress
/app/operator/evidence
/app/operator/verifications
/app/operator/devices
/app/operator/settings
```

H52 实现阶段可以先通过 redirect / route alias 兼容旧 `/operator/twin/*`，但正式导航只能指向 `/app/operator/*`。

## 4. 一级导航

### 4.1 正式侧边栏

H52 操作员端侧边栏只保留五个一级入口：

```text
地块
证据
验证
设备
设置
```

对应 route：

| 导航 | 目标 route | 状态 | 说明 |
| --- | --- | --- | --- |
| 地块 | `/app/operator/fields` | P0 | 操作员主入口。进入地块列表与地块证据孪生。 |
| 证据 | `/app/operator/evidence` | P1/P0 shell | 证据中心，按 fact / observation / state / verification 查询。 |
| 验证 | `/app/operator/verifications` | P1/P0 shell | 验证中心，优先展示 water response verification。 |
| 设备 | `/app/operator/devices` | P1 shell | 只展示操作员相关设备状态，不做 admin 配置。 |
| 设置 | `/app/operator/settings` | P1 shell | 操作员偏好与只读边界说明。 |

### 4.2 禁止出现在正式侧边栏的旧入口

以下入口不得进入 H52 正式侧边栏：

```text
Twin 总览
双子总览
Operator Twin Workbench
Field Twin
Commercial Dashboard
平台控制台
H31-H45 验收入口
Forecast 独立一级入口
Scenario 独立一级入口
Post-Irrigation 独立一级入口
Calibration 独立一级入口
ROI Ledger
Field Memory
Admin Acceptance
Dev Tools
Judge Config
Sim Config
```

这些能力如仍有价值，应以 legacy URL、二级节点、抽屉或未来专用模块存在，不得作为 H52 操作员端主入口。

## 5. 地块入口

### 5.1 `/app/operator/fields`

页面名：

```text
地块
```

职责：

- 列出操作员可访问的地块。
- 显示每块地是否有当前状态判断。
- 显示每块地证据链是否完整。
- 显示每块地是否存在水分压力闭环数据。
- 显示每块地是否存在需要补充观测的缺口。
- 进入单地块证据孪生。

必须显示字段：

- field_id
- field_name
- tenant / project / group scope
- current_state_summary
- evidence_status
- verification_status
- latest_observation_time
- gap_count
- primary_loop_status

禁止：

- 不显示 customer report 作为主卡。
- 不显示 ROI 作为主指标。
- 不按风险高低做红黄绿排序。
- 不触发任务创建。
- 不触发审批。

处理旧页面：

- 当前 `/operator/twin` 的田块列表能力迁移到 `/app/operator/fields`。
- 旧 `/operator/twin` 可保留 redirect 或 legacy URL-only。

## 6. 地块证据孪生入口

### 6.1 `/app/operator/fields/:fieldId/evidence-twin`

页面名：

```text
地块证据孪生
```

职责：

以单个 field_id 为中心展示完整证据链。

默认不以 operation_id、task_id、recommendation_id 作为入口。那些 ID 只能作为链路节点引用或详情跳转。

页面主结构：

```text
顶部摘要
→ 五层证据链
→ Water Stress Loop 摘要
→ 数据缺口与补证请求
→ 只读边界
```

顶部摘要必须回答：

- 当前地块是什么。
- 当前主要状态是什么。
- 证据链是否完整。
- 数据质量是否可用。
- 是否已有验证结果。
- 是否存在阻塞缺口。
- 当前页面是否只读。

建议字段：

- field_id
- field_name
- current_state_label
- current_state_code
- confidence_label
- evidence_status
- verification_status
- latest_update_time
- read_only_boundary

### 6.3 五层证据链

```text
RawSignal → Observation → StateEstimate → Evidence → Verification
```

- 时间

## 7. Water Stress Loop 在 IA 中的位置

### 7.1 route

Water Stress Loop 的正式 route：

```text
/app/operator/fields/:fieldId/evidence-twin/water-stress
```

它也可以在地块证据孪生主页面中作为首个闭环 tab 出现。

### 7.2 页面名

```text
水分压力闭环
```

副标题：

```text
猎鹰 1 号
```

### 7.3 定位

Water Stress Loop 是地块证据孪生下的第一条产品闭环，不是独立一级应用，不是 Forecast 页面，不是 Scenario 页面，不是 Post-Irrigation 页面。

### 7.4 节点结构

Water Stress Loop 页面使用以下 stepper：

```text
原始信号
→ 标准化观测
→ 水分压力状态
→ 水分预测
→ 灌溉情景
→ 建议候选
→ 人工审批
→ 作业计划
→ AO-ACT
→ 实执记录
→ 执行证据
→ 执行验收
→ 灌后水分响应验证
```

### 7.5 Forecast / Scenario / Post-Irrigation 的处理

以下旧页面不再作为平级主入口：

- Forecast
- Scenario Compare
- Evidence
- Post-Irrigation

迁移方式：

| 旧页面 | H52 位置 | 处理 |
| --- | --- | --- |
| `OperatorFieldTwinForecastPage.tsx` | Water Stress Loop 的“水分预测”节点 | `MIGRATE` |
| `OperatorFieldTwinScenarioComparePage.tsx` | Water Stress Loop 的“灌溉情景”节点 | `MIGRATE` |
| `OperatorFieldTwinEvidencePage.tsx` | Evidence Twin 的证据节点 / 抽屉 | `MIGRATE` |
| `OperatorFieldTwinPostIrrigationPage.tsx` | Water Stress Loop 的“灌后水分响应验证”节点 | `MIGRATE` |
| `OperatorFieldTwinCalibrationPage.tsx` | Legacy URL-only | `LEGACY_VISIBLE_BY_URL_ONLY` |

### 7.6 Water Stress Loop 禁止事项

页面不得：

- 创建 task。
- 自动审批。
- 自动派单。
- 写 ROI。
- 写 Field Memory。
- 生成 customer report。
- 为了页面完整造 scenario options。
- 把 scenario 当 task。
- 把 recommendation candidate 当 approved operation。
- 把 verification 结果写成模型已经自动学习。

## 8. 证据中心

### 8.1 `/app/operator/evidence`

页面名：

```text
证据
```

职责：

按证据对象查询，而不是按页面阶段查询。

支持查询维度：

- fact_id
- observation_id
- state_id
- estimate_id
- verification_id
- evidence_artifact_id
- device_id
- field_id
- time range
- evidence kind

默认不展示客户报告。客户报告属于 customer surface。

### 8.2 证据中心与 Evidence Twin 的关系

Evidence Twin 是以 field 为中心。

Evidence Center 是以 evidence object 为中心。

Evidence Center 中点击某个证据对象，应能跳转回：

```text
/app/operator/fields/:fieldId/evidence-twin
```

或具体节点：

```text
/app/operator/fields/:fieldId/evidence-twin/water-stress?focus=evidence&ref_id=...
```

## 14. 视觉与交互规则

### 14.1 默认视图

默认视图必须简洁，只显示：

- 当前状态
- 数据质量
- 证据链完整度
- 验证状态
- 是否需要补观测
- 最新更新时间

### 14.2 展开视图

展开后显示原始 ID、schema code、evidence refs、source refs、raw payload。

默认不把长 JSON 直接铺在页面上。

### 14.3 颜色规则

H52 P0 不使用红黄绿风险灯。

使用文字状态：

- 可用
- 受限
- 缺失
- 冲突
- 不可验证
- 需要补观测
- 已验证

### 14.4 术语规则

显示格式：

```text
中文主标签（CODE）
```

示例：

```text
水分响应已观察到（RESPONSE_OBSERVED）
中度缺水 → 正常（MODERATE_DEFICIT_TO_NORMAL）
建议候选（RECOMMENDATION_CANDIDATE）
```

### 14.5 ID 显示规则

普通视图不显示大段 ID。

需要显示的 ID 用短行展示，并提供展开。

必须保留：

- fact_id
- observation_id
- state_id / estimate_id
- verification_id
- task_id
- receipt_id
- as_executed_id
- evidence_artifact_id

## 15. 只读边界

Operator Evidence Twin P0 必须只读。

允许：

- 读取数据。
- 展开证据。
- 查看缺口。
- 查看建议候选。
- 查看审批、计划、任务、回执、实执、验收、验证。

禁止：

- 创建 recommendation。
- 创建 approval_request。
- 作出 approval_decision。
- 创建 operation_plan。
- 创建 AO-ACT task。
- 提交 receipt。
- 写 evidence_artifact。
- 写 acceptance_result。

## 16. H52.1 IA 验收标准

本 IA 被接受后，H52.1 应满足：

1. 操作员端正式产品名是“操作员证据孪生”。
2. 单地块正式页面名是“地块证据孪生”。
3. 第一条闭环是“水分压力闭环 / 猎鹰 1 号”。
4. 正式侧边栏只有：地块 / 证据 / 验证 / 设备 / 设置。
5. Forecast / Scenario / Evidence / Post-Irrigation 不再作为 operator 一级入口。
6. Forecast 是 Water Stress Loop 的预测节点。
7. Scenario 是 Water Stress Loop 的灌溉情景节点。
8. Evidence 是 Evidence Twin 的证据节点或抽屉。
9. Post-Irrigation 是 Water Stress Loop 的验证节点。
10. Calibration 不进入 P0。
11. ROI 和 Field Memory 不进入 Falcon 1。
12. 页面默认只读，不创建任务、不审批、不派单。
13. 所有旧 operator twin 页面先迁移，不直接删除。
14. `/app/operator/*` 是 H52 目标入口；实现前不得把它写成当前 runtime 已有入口。
15. 开发前还必须完成 View Model Contract 和 Water Stress Loop Acceptance Spec。

## 17. 后续文件依赖

本 IA 之后必须补齐：

```text
H52.1-EVIDENCE-TWIN-VIEW-MODEL-CONTRACT.md
H52.1-WATER-STRESS-LOOP-ACCEPTANCE.md
```

没有这两份，不应进入 React 实现。

原因：

- IA 只定义页面结构，不定义接口字段。
- Acceptance 只定义验收边界，防止前端造假。
- View Model Contract 定义前后端之间的数据契约。

## 18. 结论

H52 操作员端应从“Operator Twin / Field Twin 多页面实验”收口为：

```text
操作员证据孪生
```

它的第一条正式产品闭环是：

```text
水分压力闭环 / 猎鹰 1 号
```

正式入口应围绕地块展开，而不是围绕 forecast、scenario、evidence、post-irrigation 等工程阶段展开。

核心页面应让用户看清：

```text
这块地当前为什么被判断成这个水分状态；
用了哪些原始信号和观测；
证据质量如何；
是否形成建议候选；
是否经过审批和执行；
执行后是否被灌后响应验证。
```

这就是 H52 前端重置的产品骨架。
