本文回答以下问题：

## 1. 总体产品定位

H52 操作员端不再以 `Twin 总览`、`双子总览`、`Operator Twin Workbench` 或旧 `Workbench` 作为正式产品入口。

H52 操作员端的正式定位是：

```text
操作员证据孪生
Operator Evidence Twin
```

一句话定义：

```text
面向操作员的地块证据孪生界面，用于查看一块地从原始信号到状态判断、建议候选、执行证据和验证结果的完整链路。
```

H52 操作员端首先服务一条农业闭环：

```text
Water Stress Loop v1 / 水分压力闭环 / 猎鹰 1 号
```

它不做万能农业平台，也不做全量客户交付门户，也不做后台治理控制台。

## 2. 当前仓库现状约束

### 2.1 当前 runtime 入口

当前 runtime 主要入口仍然是：


当前 `App.tsx` 中已有 `/customer/*`、`/admin/*` shell，同时 `/operator/*` 还在 catch-all 旧 shell 下进入 operator routes。

当前 operator twin routes 是：


这些 route 当前可用，但不是 H52 的最终 canonical route。


这些命名应在 H52 实现阶段替换为：

## 3. H52 目标入口

H52 的目标 canonical entrances 是：

```text
/app/operator
/app/customer
/app/admin
```

注意：

```text
/app/* 是 H52 目标入口，不是当前 runtime 已实现入口。
```

H52 操作员端目标 route 是：

```text
/app/operator
/app/operator/fields
/app/operator/fields/:fieldId/evidence-twin
/app/operator/fields/:fieldId/evidence-twin/water-stress
/app/operator/evidence
/app/operator/verifications
/app/operator/devices
/app/operator/settings
```

H52 实现阶段可以先通过 redirect / route alias 兼容旧 `/operator/twin/*`，但正式导航只能指向 `/app/operator/*`。

## 4. 一级导航

### 4.1 正式侧边栏

H52 操作员端侧边栏只保留五个一级入口：

```text
地块
证据
验证
设备
设置
```

对应 route：

| 导航 | 目标 route | 状态 | 说明 |
| --- | --- | --- | --- |
| 地块 | `/app/operator/fields` | P0 | 操作员主入口。进入地块列表与地块证据孪生。 |
| 证据 | `/app/operator/evidence` | P1/P0 shell | 证据中心，按 fact / observation / state / verification 查询。 |
| 验证 | `/app/operator/verifications` | P1/P0 shell | 验证中心，优先展示 water response verification。 |
| 设备 | `/app/operator/devices` | P1 shell | 只展示操作员相关设备状态，不做 admin 配置。 |
| 设置 | `/app/operator/settings` | P1 shell | 操作员偏好与只读边界说明。 |

### 4.2 禁止出现在正式侧边栏的旧入口

以下入口不得进入 H52 正式侧边栏：

```text
Twin 总览
双子总览
Operator Twin Workbench
Field Twin
Commercial Dashboard
平台控制台
H31-H45 验收入口
Forecast 独立一级入口
Scenario 独立一级入口
Post-Irrigation 独立一级入口
Calibration 独立一级入口
ROI Ledger
Field Memory
Admin Acceptance
Dev Tools
Judge Config
Sim Config
```

这些能力如仍有价值，应以 legacy URL、二级节点、抽屉或未来专用模块存在，不得作为 H52 操作员端主入口。

```text

```

职责：

## 7. Water Stress Loop 在 IA 中的位置

### 7.1 route

Water Stress Loop 的正式 route：

```text
/app/operator/fields/:fieldId/evidence-twin/water-stress
```

它也可以在地块证据孪生主页面中作为首个闭环 tab 出现。

### 7.2 页面名

```text
水分压力闭环
```

副标题：

```text
猎鹰 1 号
```

### 7.3 定位

Water Stress Loop 是地块证据孪生下的第一条产品闭环，不是独立一级应用，不是 Forecast 页面，不是 Scenario 页面，不是 Post-Irrigation 页面。

### 7.4 节点结构

Water Stress Loop 页面使用以下 stepper：

```text
原始信号
→ 标准化观测
→ 水分压力状态
→ 水分预测
→ 灌溉情景
→ 建议候选
→ 人工审批
→ 作业计划
→ AO-ACT
→ 实执记录
→ 执行证据
→ 执行验收
→ 灌后水分响应验证
```

### 7.5 Forecast / Scenario / Post-Irrigation 的处理

以下旧页面不再作为平级主入口：

- Forecast
- Scenario Compare
- Evidence
- Post-Irrigation

迁移方式：

| 旧页面 | H52 位置 | 处理 |
| --- | --- | --- |
| `OperatorFieldTwinForecastPage.tsx` | Water Stress Loop 的“水分预测”节点 | `MIGRATE` |
| `OperatorFieldTwinScenarioComparePage.tsx` | Water Stress Loop 的“灌溉情景”节点 | `MIGRATE` |
| `OperatorFieldTwinEvidencePage.tsx` | Evidence Twin 的证据节点 / 抽屉 | `MIGRATE` |
| `OperatorFieldTwinPostIrrigationPage.tsx` | Water Stress Loop 的“灌后水分响应验证”节点 | `MIGRATE` |
| `OperatorFieldTwinCalibrationPage.tsx` | Legacy URL-only | `LEGACY_VISIBLE_BY_URL_ONLY` |

### 7.6 Water Stress Loop 禁止事项

页面不得：

- 创建 task。
- 自动审批。
- 自动派单。
- 写 ROI。
- 写 Field Memory。
- 生成 customer report。
- 为了页面完整造 scenario options。
- 把 scenario 当 task。
- 把 recommendation candidate 当 approved operation。
- 把 verification 结果写成模型已经自动学习。

## 12. 冲突页面与旧入口治理优先级

H52 实现前，不应优先删除旧 operator 页面。

旧 operator 页面里有已验证过的 API 映射、字段展示、证据链卡片和验收经验。直接删除会导致两类风险：

1. 已完成后端能力失去可视化参考。
2. 无法确认被删除的是废弃页面，还是仍承担验收或迁移价值的页面。

H52 的正确治理顺序是：

```text
先修正会误判路由行为的结构性问题
→ 再降级旧 operator 页面为 URL-only
→ 新页面稳定后再删除旧实现
```

---

### 12.1 P0：必须先处理的结构性冲突

P0 对象不是因为它们“最旧”，而是因为它们会继续误导后续路由、入口、验收和页面行为判断。

这些对象必须优先处理，否则 H52 后续开发会继续混乱。

| 对象 | 问题类型 | 处理意见 | 原因 |
| --- | --- | --- | --- |
| `CustomerLayout` pathname override | `layout override` | `MODIFY / REMOVE_OVERRIDE` | route children 已经由路由表传入，但 layout 内部又根据 `location.pathname` 替换页面。只改 route table 不一定改变真实渲染结果。 |
| `/customer/fields` 双实现 | `layout override / duplicate implementation` | `MODIFY / SINGLE_OWNER` | customer fields 同时可能由 route table 与 CustomerLayout 内部替换控制。必须确定唯一页面 owner。 |
| `/customer/operations` 双实现 | `layout override / duplicate implementation` | `MODIFY / SINGLE_OWNER` | customer operations 同时存在 route 层和 layout 层实现，后续迁移容易误判真实渲染源。 |
| `/customer/reports` 双实现 | `layout override / duplicate implementation` | `MODIFY / SINGLE_OWNER` | customer reports 是客户交付入口，不能由两个实现并存。 |
| `/skills/runs` | `route parameter contract risk` | `BUG_RISK / ACCEPTANCE_REQUIRED` | `/skills/runs` 与 `/skills/runs/:runId` 指向同一 detail page 时，前者没有 `runId` 参数，存在运行时契约风险。 |
| `SubmitScenarioToRecommendationPanel` | `write-entry boundary risk` | `REMOVE_FROM_P0_SURFACE / LEGACY_OR_FUTURE_WRITE_SURFACE_ONLY` | H52.1 Operator Evidence Twin P0 必须只读。该组件涉及 scenario → recommendation candidate 的写入口，不得出现在只读 Evidence Twin / Water Stress Loop P0 页面。 |

#### 12.1.1 CustomerLayout pathname override

处理规则：

```text
Customer route 的真实页面 owner 必须只有一个。
```

允许：

- `CustomerShell` / route table 负责页面选择。
- `CustomerLayout` 只负责布局。

禁止：

- `CustomerLayout` 根据 `location.pathname` 自行替换 children。
- route table 和 layout 同时决定 `/customer/fields`、`/customer/operations`、`/customer/reports` 的页面实现。

处理结果目标：

```text
/customer/fields      → single route owner
/customer/operations  → single route owner
/customer/reports     → single route owner
```

#### 12.1.2 `/skills/runs` 参数契约问题

处理规则：

```text
无 runId 的列表页不能复用需要 runId 的详情页。
```

允许：

- `/skills/runs` 指向 SkillRunListPage。
- `/skills/runs/:runId` 指向 SkillRunDetailPage。

禁止：

- `/skills/runs` 和 `/skills/runs/:runId` 同时指向需要 `runId` 的 detail page。

处理前必须补 acceptance：

- 访问 `/skills/runs` 不应抛出缺少 runId 的错误。
- 访问 `/skills/runs/:runId` 才进入详情页。
- legacy route 若保留，必须标明不是 H52 operator P0 入口。

#### 12.1.3 SubmitScenarioToRecommendationPanel 写入口

处理规则：

```text
H52.1 Operator Evidence Twin P0 只读。
```

因此：

- Water Stress Loop P0 可以展示 recommendation candidate。
- 可以展示 `human_approval_required = true`。
- 可以展示 `no_direct_execution = true`。
- 不得在 P0 页面提供提交 recommendation 的写按钮。
- 不得从 P0 页面创建 approval。
- 不得从 P0 页面创建 operation plan。
- 不得从 P0 页面创建 AO-ACT task。

处理意见：

```text
SubmitScenarioToRecommendationPanel:
```

### 12.3 P2：新页面完成后再删除

以下对象不能现在删除。必须等 H52 新页面、route alias、验收脚本、反向引用检查完成后，才能进入删除候选。

| 对象 | 处理意见 | 删除前条件 |
| --- | --- | --- |
| 旧 Operator Twin 命名页面 | `DELETE_CANDIDATE_AFTER_MIGRATION` | 新 Evidence Twin 页面覆盖其展示能力；旧 route 有 redirect 或 legacy policy；验收通过。 |
| 旧 root `/dashboard` | `DELETE_CANDIDATE_AFTER_SHELL_RESET` | customer/admin/operator 新 shell 完成；无正式导航引用。 |
| 旧 root `/fields` | `DELETE_CANDIDATE_AFTER_SHELL_RESET` | `/app/operator/fields` 与 customer/admin field surface 完成。 |
| 旧 root `/operations` | `DELETE_CANDIDATE_AFTER_SHELL_RESET` | operator execution / customer operation report / admin operation view 分离完成。 |
| 旧 root `/devices` | `DELETE_CANDIDATE_AFTER_SHELL_RESET` | operator devices 与 admin devices 分离完成。 |
| 旧 AppShell / SideNav 依赖 | `DELETE_CANDIDATE_AFTER_NEW_SHELLS` | `/app/operator`、`/app/customer`、`/app/admin` 新 shell 稳定后再删。 |
| 旧 route modules 里的重复 customer/admin route | `DELETE_CANDIDATE_AFTER_ROUTE_OWNER_FREEZE` | CustomerShell / AdminShell 页面 owner 冻结；无反向引用；验收通过。 |
| debug/dev routes | `DELETE_CANDIDATE_AFTER_DEV_POLICY` | 明确 dev-only policy；确认无验收脚本依赖。 |

删除前必须满足：

1. 新 route 已存在。
2. 新 route 验收通过。
3. 旧 route 没有正式导航引用。
4. 旧 route 没有 acceptance 脚本依赖。
5. 旧 route 没有 customer/operator/admin runtime 依赖。
6. 已记录 redirect / legacy policy。
7. 已完成反向引用搜索。

### 12.4 总体处理顺序

H52 不按“旧页面先删”的顺序推进。

正式顺序如下：

```text
P0：修复真实路由行为判断问题
  - CustomerLayout pathname override
  - customer 双实现
  - /skills/runs 参数契约
  - P0 页面移除 SubmitScenarioToRecommendationPanel 写入口

P1：旧 operator 页面降级
  - 从正式导航撤下
  - 保留 URL-only
  - 标记 Legacy

P2：新页面完成后删除旧实现
  - 删除旧 Operator Twin 命名页面
  - 删除旧 root routes
  - 删除旧 AppShell / SideNav 依赖
  - 删除重复 route modules
```

结论：

```text
H52 的第一步不是删除旧页面。
H52 的第一步是消除会误判 route 行为的结构性冲突。
```

## 15. 视觉与交互规则

### 15.1 默认视图

默认视图必须简洁，只显示：

- 当前状态
- 数据质量
- 证据链完整度
- 验证状态
- 是否需要补观测
- 最新更新时间

### 15.2 展开视图

展开后显示原始 ID、schema code、evidence refs、source refs、raw payload。

默认不把长 JSON 直接铺在页面上。

### 15.3 颜色规则

H52 P0 不使用红黄绿风险灯。

使用文字状态：

- 可用
- 受限
- 缺失
- 冲突
- 不可验证
- 需要补观测
- 已验证

### 15.4 术语规则

显示格式：

```text
中文主标签（CODE）
```

示例：

```text
水分响应已观察到（RESPONSE_OBSERVED）
中度缺水 → 正常（MODERATE_DEFICIT_TO_NORMAL）
建议候选（RECOMMENDATION_CANDIDATE）
```

### 15.5 ID 显示规则

普通视图不显示大段 ID。

需要显示的 ID 用短行展示，并提供展开。

必须保留：

- fact_id
- observation_id
- state_id / estimate_id
- verification_id
- task_id
- receipt_id
- as_executed_id
- evidence_artifact_id

## 16. 只读边界

允许：

- 作出 approval_decision。
- 创建 operation_plan。
- 创建 AO-ACT task。
- 提交 receipt。
- 写 evidence_artifact。
- 写 acceptance_result。
- 写 water_response_verification。
- 写 Field Memory。
- 写 ROI。
- 生成 customer report。

若后续需要写动作，必须进入单独 Sprint 和单独 acceptance。

## 17. H52.1 IA 验收标准

本 IA 被接受后，H52.1 应满足：

1. 操作员端正式产品名是“操作员证据孪生”。
2. 单地块正式页面名是“地块证据孪生”。
3. 第一条闭环是“水分压力闭环 / 猎鹰 1 号”。
4. 正式侧边栏只有：地块 / 证据 / 验证 / 设备 / 设置。
5. Forecast / Scenario / Evidence / Post-Irrigation 不再作为 operator 一级入口。
6. Forecast 是 Water Stress Loop 的预测节点。
7. Scenario 是 Water Stress Loop 的灌溉情景节点。
8. Evidence 是 Evidence Twin 的证据节点或抽屉。
9. Post-Irrigation 是 Water Stress Loop 的验证节点。
10. Calibration 不进入 P0。
11. ROI 和 Field Memory 不进入 Falcon 1。
12. 页面默认只读，不创建任务、不审批、不派单。
13. 所有旧 operator twin 页面先迁移，不直接删除。
14. `/app/operator/*` 是 H52 目标入口；实现前不得把它写成当前 runtime 已有入口。
15. 开发前还必须完成 View Model Contract 和 Water Stress Loop Acceptance Spec。

## 18. 后续文件依赖

本 IA 之后必须补齐：

```text
H52.1-EVIDENCE-TWIN-VIEW-MODEL-CONTRACT.md
H52.1-WATER-STRESS-LOOP-ACCEPTANCE.md
```

没有这两份，不应进入 React 实现。

- IA 只定义页面结构，不定义接口字段。
- Acceptance 只定义验收边界，防止前端造假。
- View Model Contract 定义前后端之间的数据契约。

## 19. 结论

H52 操作员端应从“Operator Twin / Field Twin 多页面实验”收口为：

```text
操作员证据孪生
```

它的第一条正式产品闭环是：

```text
水分压力闭环 / 猎鹰 1 号
```

正式入口应围绕地块展开，而不是围绕 forecast、scenario、evidence、post-irrigation 等工程阶段展开。

核心页面应让用户看清：

```text
这块地当前为什么被判断成这个水分状态；
用了哪些原始信号和观测；
证据质量如何；
是否形成建议候选；
是否经过审批和执行；
执行后是否被灌后响应验证。
```

这就是 H52 前端重置的产品骨架。
