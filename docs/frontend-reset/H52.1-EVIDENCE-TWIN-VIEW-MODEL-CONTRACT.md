## 0. 目的

本文冻结 H52 Operator Evidence Twin 的前端视图模型契约。

它解决一个核心问题：

```text
前端不再由每个页面分别拼多个旧接口、旧 route、旧卡片和旧字段。
```

H52 之后，操作员端应围绕一个统一的 Evidence Twin View Model 展示地块证据链和 Water Stress Loop，而不是继续让 Workspace、Forecast、Scenario、Evidence、Post-Irrigation 页面各自定义自己的数据形状。

本文不要求立即实现单一后端接口，但它定义前端应消费的目标 contract。当前多个旧接口可以先由前端 adapter 或后端 read model 聚合到该 contract。

## 2. 目标接口

目标 route 建议：

```text
GET /api/v1/operator/fields/:fieldId/evidence-twin
```

Water Stress Loop 可作为同一 view model 下的 primary loop：

```text
GET /api/v1/operator/fields/:fieldId/evidence-twin?loop=water-stress
```

如果后端暂时不实现该单一接口，前端 adapter 可以从下列现有接口聚合：

```text
GET /api/v1/operator/twin/source-indexes
GET /api/v1/operator/twin
GET /api/v1/operator/twin/fields/:fieldId
GET /api/v1/operator/twin/fields/:fieldId/forecast
GET /api/v1/operator/twin/fields/:fieldId/scenarios
GET /api/v1/operator/twin/fields/:fieldId/evidence
GET /api/v1/operator/twin/fields/:fieldId/post-irrigation
GET /api/v1/operator/twin/fields/:fieldId/h31-h45-closure
```

目标前端 route：

```text
/app/operator/fields/:fieldId/evidence-twin
/app/operator/fields/:fieldId/evidence-twin/water-stress
```

注意：

```text
/app/* 是 H52 目标入口，不是当前 runtime 已实现入口。
```

## 5. 通用节点契约

所有 lineage 节点、Water Stress Loop 节点和 conflict policy 节点都必须采用相同的最小字段。

```ts
type EvidenceTwinNodeV1 = {
  id: string;
  label: string;
  kind: string;
  schema_ref: string | null;
  status: "AVAILABLE" | "LIMITED" | "MISSING" | "BLOCKING" | "NOT_APPLICABLE" | string;
  time: {
    occurred_at: string | null;
    observed_at: string | null;
    computed_at: string | null;
    updated_at: string | null;
    latest_ts_ms: number | null;
  };
  quality: {
    status: "AVAILABLE" | "LIMITED" | "BLOCKING" | "UNKNOWN" | string;
    quality_flags: string[];
    blocking_reasons: string[];
    confidence_penalty: string | null;
  };
  confidence: {
    label: string | null;
    score: number | null;
    level: string | null;
  };
  source_refs: EvidenceTwinRefV1[];
  evidence_refs: EvidenceTwinRefV1[];
  expand_payload: Record<string, unknown> | null;
  ui_policy: {
    default_collapsed: boolean;
    show_raw_payload: boolean;
    show_internal_ids: boolean;
    show_customer_safe_label: boolean;
  };
  write_policy: {
    write_ready: false;
    allowed_actions: [];
  };
};
```

强制规则：

- `id` 不得为空。缺失真实 ID 时使用稳定 synthetic id，例如 `missing:water_stress_state`。
- `label` 使用中文主标签。
- `kind` 使用产品级类别，例如 `raw_signal`、`observation`、`state_estimate`、`evidence`、`verification`、`scenario`。
- `schema_ref` 使用仓库事实类型、index 表或 projection 名称。
- `expand_payload` 可以包含原始字段，但默认折叠。
- `write_policy.write_ready` 在 H52.1 P0 必须为 `false`。

## 10. Water Stress Loop 契约

### 10.1 steps

```ts
type WaterStressLoopStepV1 = EvidenceTwinNodeV1 & {
  step_code:
    | "RAW_SIGNAL"
    | "OBSERVATION"
    | "WATER_STRESS_STATE"
    | "FORECAST"
    | "SCENARIO"
    | "RECOMMENDATION"
    | "APPROVAL"
    | "OPERATION_PLAN"
    | "AO_ACT"
    | "AS_EXECUTED"
    | "EVIDENCE"
    | "ACCEPTANCE"
    | "VERIFICATION";
  order: number;
  required_for_p0: boolean;
};
```

固定顺序：

```text
原始信号
→ 标准化观测
→ 水分压力状态
→ 水分预测
→ 灌溉情景
→ 建议候选
→ 人工审批
→ 作业计划
→ AO-ACT
→ 实执记录
→ 执行证据
→ 执行验收
→ 灌后水分响应验证
```

### 10.2 inputs

```ts
type WaterStressInputsV1 = {
  soil_moisture: EvidenceTwinNodeV1[];
  canopy_temperature: EvidenceTwinNodeV1[];
  weather_forecast: EvidenceTwinNodeV1[];
  irrigation_event: EvidenceTwinNodeV1[];
};
```

处理规则：

- `soil_moisture` 可以来自 raw telemetry、device observation、soil moisture sensing window。
- `canopy_temperature` 如果未正式并入主闭环，应标记 `LIMITED`，作为辅助压力信号。
- `weather_forecast` 区分外部天气输入与根区水预测。
- `irrigation_event` 区分 scenario event 与 executed event。

## 15. 旧页面与冲突实现处理意见

View Model Contract 必须比旧页面更高一级。旧页面可以消费 contract，但不能继续各自定义 contract。

### 15.1 `OperatorFieldTwinWorkspacePage.tsx`

当前价值：

- 已聚合 workspace 与 H31-H45 closure。
- 已包含 scope policy、state vector、data coverage、evidence summary、data gaps、layers、scenario boundary、recommendation candidate、decision-to-water-response chain、read-only boundary。

处理意见：

```text
MIGRATE
```

迁移规则：

- 其卡片能力迁入 Evidence Twin 主页面。
- 其旧 `Fact / Estimate / Forecast / Scenario / Recommendation` layer 只能作为 adapter 输入。
- 其 route `/operator/twin/fields/:fieldId` 不作为新 canonical route。
- 其 hero action 中的旧 `/operator/twin/...` 链接应迁移到 `/app/operator/...`。

### 15.2 `OperatorFieldTwinForecastPage.tsx`

处理意见：

```text
MIGRATE_TO_WATER_STRESS_FORECAST_NODE
```

不得作为 operator 一级入口。

### 15.3 `OperatorFieldTwinScenarioComparePage.tsx`

当前问题：

- 它展示 scenario compare。
- 它引用 `SubmitScenarioToRecommendationPanel`。
- 因此它存在 P0 只读边界风险。

处理意见：

```text
MIGRATE_READ_PART_ONLY
REMOVE_WRITE_PANEL_FROM_P0_SURFACE
```

规则：

- `ScenarioCompareTable` 可迁移。
- `ScenarioStatusCard` 可迁移。
- `ScenarioBoundaryCard` 可迁移。
- `SubmitScenarioToRecommendationPanel` 不进入 H52.1 P0。

### 15.4 `SubmitScenarioToRecommendationPanel.tsx`

处理意见：

```text
LEGACY_OR_FUTURE_WRITE_SURFACE_ONLY
```

原因：

- 该组件可以提交 scenario option 到 recommendation。
- 即使它不审批、不派单、不创建 operation plan、不创建 AO-ACT task，它仍然是写入口。
- H52.1 Operator Evidence Twin P0 必须只读。

Contract 处理：

- View model 可以展示 `recommendation_candidate`。
- View model 不暴露 submit action。
- `allowed_actions` 必须为空数组。
- `writeReady=false`。

### 15.5 `OperatorFieldTwinEvidencePage.tsx`

处理意见：

```text
MIGRATE_TO_EVIDENCE_NODE_OR_DRAWER
```

不得继续作为孤立工程阶段页。

### 15.6 `OperatorFieldTwinPostIrrigationPage.tsx`

处理意见：

```text
MIGRATE_TO_WATER_RESPONSE_VERIFICATION_NODE
```

它是 Water Stress Loop 的验证节点，不是独立一级入口。

### 15.7 `OperatorFieldTwinCalibrationPage.tsx`

处理意见：

```text
LEGACY_VISIBLE_BY_URL_ONLY
```

Calibration 不进入 H52.1 P0。

### 15.8 `CustomerLayout` pathname override

处理意见：

```text
MODIFY / REMOVE_OVERRIDE
```

原因：

- Customer route 的真实页面 owner 必须只有一个。
- `CustomerLayout` 当前会根据 pathname 替换 children。
- 这会导致只改 route table 不一定改变真实渲染结果。

虽然该问题不属于 Operator Evidence Twin View Model 本体，但它必须在 H52 route governance 中作为 P0 处理，否则 customer surface 的 contract owner 会继续混乱。

### 15.9 `/skills/runs` 参数契约问题

处理意见：

```text
BUG_RISK / ACCEPTANCE_REQUIRED
```

原因：

- `/skills/runs` 不应复用需要 `runId` 的详情页。
- `/skills/runs` 应为 list page。
- `/skills/runs/:runId` 才是 detail page。

该问题不进入 Operator Evidence Twin P0，但属于 route governance P0。

### 15.10 旧 root routes 与 AppShell / SideNav

处理意见：

```text
DELETE_CANDIDATE_AFTER_NEW_SHELLS
```

删除前条件：

- `/app/operator` 新 shell 完成。
- `/app/customer` 新 shell 完成。
- `/app/admin` 新 shell 完成。
- 无正式导航引用。
- 无 acceptance 脚本依赖。
- 已记录 redirect / legacy policy。

## 18. 验收标准

本 contract 被接受后，必须满足：

1. 前端有唯一 Evidence Twin view model 目标结构。
2. 每个节点都有 `id / label / kind / status / time / quality / confidence / source_refs / evidence_refs / expand_payload`。
3. View model 显式包含 RawSignal / Observation / StateEstimate / Evidence / Verification 五层。
4. View model 显式包含 Water Stress Loop。
5. Forecast 与 Scenario 是 Water Stress Loop 节点，不是 operator 一级入口。
6. Recommendation 只能展示 candidate，不提供提交按钮。
7. `writeReady=false`。
8. `taskCreationReady=false`。
9. `memoryWriteReady=false`。
10. `roiWriteReady=false`。
11. scenario options 为空时显示 gap，不造假。
12. root-zone forecast 缺失时显示 gap，不造假。
13. canopy temperature 未进入主闭环时显示为 auxiliary / limited signal。
14. H31-H45 closure 可以映射到 steps / execution_tail / verification。
15. 旧 operator pages 只能作为 adapter source 或 legacy URL-only，不再定义新 contract。
16. `SubmitScenarioToRecommendationPanel` 不进入 H52.1 P0 正式 surface。
17. `CustomerLayout` override、customer 双实现、`/skills/runs` 参数问题必须作为 route governance P0，不得被本 contract 掩盖。
18. 本 contract 不要求立即新增事实类型。
19. 本 contract 不要求立即新增 DB 表。
20. 本 contract 不触发任何运行时代码改动。

---

## 19. 结论

H52.1 Evidence Twin View Model Contract 的核心不是新增一个漂亮 JSON。

它的核心是把当前分散的 Operator Twin 页面、H31-H45 closure、forecast panel、scenario compare、evidence quality、post-irrigation verification 统一收口为一个前端可消费的证据孪生对象。

正式原则：

```text
旧页面可以迁移。
旧接口可以适配。
旧写入口不能进入 P0。
旧 route 不能定义新 contract。
```

H52 前端开发必须以本 contract 为数据边界。
